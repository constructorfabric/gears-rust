## 1. Overview

### 1.1 Purpose

Licensing Service provides a canonical platform API and SDK for licensing and quota decisions across ModKit and
CyberFabric modules. In p1, the contract is intentionally narrow: authority routing, entitlement lookup sufficient for
online checks, basic quota enforcement, simplified reservation lifecycle, and durable accounting for the minimal
runtime surface `Check`, `CheckAndReserve`, and `Commit`.

The module exists to give platform modules one stable interface for licensing and quota concerns regardless of whether
the authoritative licensing source is native ModKit licensing or a legacy VendorA licensing deployment. Licensing
Service owns the integrated decision boundary. Richer explanation, reconciliation, advanced policy resolution, and
broader runtime surfaces are later-tier extensions rather than part of the p1 contract.

This PRD states the p1 delivery contract first and reserves p2/p3 for later operational and audit-oriented
expansion.

### 1.2 Background / Problem Statement

The platform needs a licensing capability that supports both new ModKit-native modules and deployments that run inside
legacy VendorA data center environments. Some product scopes are already governed by VendorA licensing systems that own
commercial truth for subscriptions, offering items, usage constraints, and entitlement lifecycle.

Without a canonical licensing facade, every consuming module would need to integrate with legacy licensing directly or
invent local feature-gating and quota interpretation rules. This would create duplicated integration logic, inconsistent
entitlement semantics, and no clean migration path toward a native platform licensing model.

The platform also needs runtime quota enforcement for workloads that consume constrained resources such as requests,
tokens, storage, cost units, and concurrency slots. Without a reusable quota module, each consumer implements its own
local counters, ad hoc rules, and inconsistent error handling. This creates drift in policy semantics, weak
auditability, and unclear operator visibility into why a request was accepted, downgraded, or rejected.

Entitlement truth and runtime quota enforcement are tightly coupled: every quota decision starts with an entitlement
check and an effective limit lookup. Separating them into independent services forces consumers to orchestrate two
calls for one logical decision or forces one service to orchestrate the other. In a modular monolith, internal layering
within one module gives the same separation of concerns without the coordination overhead of a service boundary.
This PRD therefore contracts the minimum integrated decision boundary first and defers broader behavior to later
tiers.

### 1.3 Goals (Business Outcomes)

#### Immediate goals (p1 contract)

- Provide one canonical licensing and quota API and SDK for all ModKit and CyberFabric modules.
- Support federated authority so legacy VendorA licensing can remain authoritative for selected scopes during
  migration.
- Normalize legacy and native licensing concepts enough to support stable online checks and basic enforcement.
- Expose feature entitlement and effective-limit data without forcing consumers to understand legacy domain quirks.
- Eliminate inconsistent per-module quota implementations by providing one platform module for policy, decisioning,
  reservation, and durable accounting in the minimal p1 runtime path.
- Protect platform cost, capacity, and service quality for AI and non-AI workloads through auditable p1 runtime
  enforcement and durable usage tracking.

#### Future evolution goals (p2/p3)

- Expand the runtime surface with release, effective quota inspection, usage summaries, and other operational tools.
- Add reconciliation, richer explanation, and stricter idempotency semantics without changing the core integration
  boundary.
- Enable multi-dimensional, hierarchical, and more deterministic quota behavior as later-tier capabilities.
- Strengthen migration readiness through shadow comparison, deeper observability, and eventual native-authoritative
  rollout.

### 1.4 Implementation Priorities

This PRD uses priority-tiered delivery. p1 defines the minimum enforceable contract; p2 and p3 add operational maturity,
determinism, and deeper auditability.

#### Priority p1 (Minimal Viable Enforcement)

Scope:

- canonical licensing facade and authority routing
- entitlement read model sufficient for online checks
- basic quota enforcement
- minimal decision contract
- simplified reservation lifecycle
- basic key-based idempotency
- minimal runtime API surface: `Check`, `CheckAndReserve`, `Commit`

Out of scope:

- canonical request equality
- byte-identical cross-node determinism
- full reconciliation semantics
- advanced hierarchical policy resolution
- full explanation graph
- shadow-comparison guarantees
- lease lifecycle and direct update flows

#### Priority p2 (Operational Maturity)

Scope:

- `released` lifecycle semantics
- reconciliation workflows and usage-summary views
- stricter idempotency matching
- scoped policy resolution and basic explanation metadata
- expanded runtime surface (`Release`, `GetEffectiveQuota`, `GetUsageSummary`, optional `Update`)
- external-authority operational visibility

#### Priority p3 (Deterministic & Auditable System)

Scope:

- canonical determinism across replay, audit, and comparison
- full explanation graph and structured attribution
- hierarchical / expanded policy resolution
- shadow comparison guarantees
- lease-based concurrency lifecycle
- advanced auditability and recovery semantics

Interpretation rules:

- `Priority` on an FR identifies the earliest delivery priority tier in which that requirement becomes part of the
  delivery contract.
- `Strictness = REQUIRED` means the relevant priority exit depends on implementing the requirement as written for that
  tier.
- `Strictness = RELAXED` means the requirement is a target-state constraint; lower-priority tiers may implement an
  intentionally simpler form documented in §4.6.
- `Priority` on an NFR identifies the earliest tier at which the NFR is **tracked**, not necessarily the tier
  at which it becomes an exit criterion. An NFR is an exit criterion at priority tier `pN` only when the FRs
  it supports are themselves in scope at `pN`; when a supported FR lands at a later tier, the NFR becomes an
  exit criterion at that later tier. NFRs whose body explicitly defers to a later tier (for example,
  `cpt-cf-licensing-service-nfr-reconciliation-lag`) follow that deferred wording and are **not** p1 exit
  criteria despite carrying a `p1` tag.

## 2. Actors

> **Note**: Stakeholder needs are managed at project/task level by steering committee. Document **actors** (users,
> systems) that interact with this module.

### 2.1 Human Actors

#### Platform Operator

**ID**: `cpt-cf-licensing-service-actor-platform-operator`

- **Role**: Operates platform services and configures authority routing, rollout modes, freshness policy, quota policy
  defaults, emergency overrides, observability thresholds, and investigates degraded or mismatched decisions.
- **Needs**: Deterministic policy administration, clear decision explanations, safe degradation behavior, reliable
  visibility into authority status, projection freshness, quota usage, and reconciliation health.

#### Tenant Administrator

**ID**: `cpt-cf-licensing-service-actor-tenant-administrator`

- **Role**: Reviews tenant entitlements and effective limits, manages tenant-level quota allocations and override
  policies, and interprets why capabilities are enabled, denied, or limited.
- **Needs**: Predictable tenant-scoped limits, accurate remaining quota visibility, clear entitlement visibility, and
  explainable outcomes.

#### Migration Engineer

**ID**: `cpt-cf-licensing-service-actor-migration-engineer`

- **Role**: Validates parity between authoritative legacy decisions and shadow native decisions during migration.
- **Needs**: Shadow comparison output, mismatch reporting, and rollout safety signals.

#### Service Integrator

**ID**: `cpt-cf-licensing-service-actor-service-integrator`

- **Role**: Integrates the Licensing Service SDK into consumer modules such as gateways, workers, and schedulers.
- **Needs**: Stable decision and settlement interfaces, idempotent lifecycle semantics, and generic support for multiple
  resource types and workload classes.

### 2.2 System Actors

#### Consumer Module

**ID**: `cpt-cf-licensing-service-actor-consumer-module`

- **Role**: A runtime client such as an API gateway, LLM service, upload service, job scheduler, platform module, or
  worker that requests licensing decisions, quota decisions, and settles usage through the canonical SDK or API.

#### Control Plane

**ID**: `cpt-cf-licensing-service-actor-control-plane`

- **Role**: Administrative system that manages policies, queries effective licensing and quota state, and exposes
  visibility to operators and tenant administrators.

#### Reconciliation Worker

**ID**: `cpt-cf-licensing-service-actor-reconciliation-worker`

- **Role**: Background processor that expires leaked reservations, reconciles mismatches, and maintains aggregate usage
  views.

#### Legacy VendorA Licensing System

**ID**: `cpt-cf-licensing-service-actor-legacy-vendora-licensing`

- **Role**: Acts as the authoritative commercial source for legacy-controlled scopes and supplies source data through
  the legacy adapter.

#### Native Licensing Store

**ID**: `cpt-cf-licensing-service-actor-native-licensing-store`

- **Role**: Persists native-authoritative entitlement grants, offering-item associations, and effective limits for
  platform-native scopes.

## 3. Operational Concept & Environment

> **Note**: Project-wide runtime, OS, architecture, lifecycle policy, and integration patterns defined in root PRD.
> Document only module-specific deviations here. **Delete this section if no special constraints.**

### 3.1 Module-Specific Environment Constraints

- The module must support deployments both inside legacy VendorA data center environments and in platform-native
  environments.
- Hot-path licensing decisions must rely on local projections, caches, or native stores rather than deep synchronous
  traversal of legacy authority graphs.
- Degraded behavior must preserve explicit authority identity, freshness state, and limiting-factor information when
  upstream authority data is stale or unavailable.
- The module operates in a multi-tenant platform environment and every runtime decision request includes subject, scope,
  action, resource dimensions, and idempotency metadata.
- The module requires a low-latency enforcement path for preview and reservation decisions and a separate durable path
  for authoritative accounting and reconciliation.
- The module must support both in-process and out-of-process consumer integrations because enforcement happens near
  execution points across gateways, services, workers, and background processors.

## 4. Scope

### 4.1 In Scope

- Canonical licensing and quota API and SDK for platform consumers.
- Authority routing between legacy-authoritative and native-authoritative scopes.
- Canonical read model for feature entitlements, offering items, and effective limits.
- Legacy VendorA licensing adapter as a ModKit plugin boundary behind the Licensing Service public API and normalization
  boundary.
- Native licensing authority mode for platform-native scopes.
- Decision explanation that identifies authority, limiting factor, and policy basis.
- Shadow-mode comparison support during migration.
- Licensing-health observability, reconciliation hooks, and sync status visibility.
- Quota policy administration for default plans, scoped policies, overrides, and expiration.
- Deterministic policy resolution across platform, region, tenant, and subject scopes, with product, offering,
  deployment, and usage-dimension filters carried by the normalized licensing scope.
- Multi-dimensional quota evaluation across requests, tokens, storage, cost units, concurrency, and extensible custom
  resource types.
- Soft and hard quota behaviors including warning, downgrade, capping, reject, and retry-after responses.
- Reservation-based enforcement flows with estimated and actual usage settlement.
- Durable usage accounting, aggregate usage summaries, and reconciliation of reservation drift.
- Explainable decision responses for preview, reserve, effective quota, and usage visibility operations.
- AI and LLM workload quota coverage including request, token, budget, concurrency, and product- or
  offering-scoped governance without AI-specific global scope levels.
- Reusable support for non-AI consumers such as uploads, storage, vector operations, and job execution.

### 4.2 Out of Scope

- License key generation and serial-number pool operations; owned by legacy VendorA licensing or a future dedicated
  licensing backoffice.
- Billing, invoicing, taxation, and payment processing; owned by billing and commercial systems.
- Direct per-module legacy API integrations; forbidden by architecture because modules must use the licensing-service
  SDK.
- Detailed reseller, distributor, OEM, and perpetual-license backoffice workflows unless explicitly adopted later.
- Generic feature flag management unrelated to licensing or entitlements.
- A mandatory single inline choke point through which all system traffic must synchronously pass.
- Provider-specific quota logic that only works for one LLM vendor or one product module.
- Arbitrary business pricing policy beyond quota enforcement and usage accounting boundaries.
- Full global strong-consistency guarantees with zero latency, zero overshoot, and infinite scale in the same operating
  mode.

### 4.3 Unified Module Rationale

#### Clock authority

Every PRD rule that compares a timestamp or duration against a threshold (for example, `emergency_override.expires_at`,
`reservation.expires_at`, `lease.expires_at`, `window_end`, `idempotency_record` age) **MUST** follow the boundary-
consistency rules below. The module does **not** assume an externally provided distributed clock service; the rules
are expressible on top of (a) the fast-path state store as the single writer for record-creation timestamps and
(b) per-node wall-clock reads bounded by `allowed_clock_skew_seconds`.

1. **Persisted boundary timestamps are authoritative for record age and terminal age.** `created_at`, `accepted_at`,
   `first_acquired_at`, `committed_at`, `released_at`, and `expired_at` are written exactly once by the fast-path
   state store when the corresponding creation or terminal transition is accepted. Consuming nodes **MUST NOT**
   recompute record age or terminal age against their own wall-clock. `expired_at` records the first accepted
   transition that classifies the record as `expired`; it is **NOT** recomputed from `expires_at` by readers.
   This realises the single-writer invariant for `created_at` already stated in
   `cpt-cf-licensing-service-fr-idempotency-vs-archive` and extends it uniformly to every record-age boundary in
   this PRD.
2. **"Now" comparisons read the local module wall-clock with bounded skew.** For boundaries that require comparing
   a persisted timestamp or derived `expires_at` against "now" (reservation/lease expiry, window attribution,
   registry retirement, resource-type retirement, freshness-threshold classification, emergency-override
   applicability), the evaluating node samples `now_at_evaluation` once from its own wall-clock and reuses that
   value throughout the evaluation. The same rule applies to admin-mutation validation requests. The evaluating
   node **MUST** apply conservative skew handling: if the computed excess beyond the relevant threshold is
   `<= allowed_clock_skew_seconds`, the comparison remains in the less-severe class (`fresh` over `stale`,
   `stale_within_policy` over `stale_beyond_policy`, `valid` over `expired`, `non-retired` over `retired`). Only
   an excess `> allowed_clock_skew_seconds` may cross into the more-severe class, which then falls into the
   fail-closed branch (degraded / stale-beyond-policy / rejected) and emits a `clock_drift` telemetry warning
   tagged with the normalized licensing scope. Two concurrent nodes therefore agree on every boundary
   classification **except** within a window of size `allowed_clock_skew_seconds` centred on the boundary instant;
   within that window either classification is permissible and **MUST NOT** be treated as an implementation
   divergence during shadow comparison or audit replay.
3. **Derived `expires_at` uses the writer's wall-clock at acceptance.** When a record's `expires_at` is derived
   (`accepted_at + ttl`), the derivation is performed on the writing node using its wall-clock at acceptance and
   then persisted; consuming nodes read the persisted `expires_at` rather than recomputing. This keeps the
   `expires_at` value consistent with rule (1).
4. **Emergency-override and window boundaries are evaluated against `now_at_module`** per rule (2); no external
   clock service is required. Shadow comparison that straddles the skew window **MUST NOT** be reported as a
   correctness mismatch.

This rule applies uniformly to entitlement-lifecycle expiry, reservation/lease expiry, window attribution,
emergency-override applicability, and idempotency-record expiry. Within the `allowed_clock_skew_seconds` window
at any boundary, consumers **MUST** tolerate a bounded amount of transient classification disagreement between
nodes; outside that window, the classification is deterministic by construction.

#### Consistency model

The fast-path state store is the **single linearization point** for every state-changing operation
(`CheckAndReserve`, `Commit`, `Release`, `AcquireLease`, `RenewLease`, `ReleaseLease`, idempotency-record
writes, and per-window counter mutations). The module **MUST** provide the following client-observable
consistency guarantees on top of that single writer:

1. **Read-your-writes on the fast-path object graph.** Once a state-changing operation returns success
   (not a service error, and not a replayed result from idempotency match), every subsequent admission
   decision, `Commit`, `Release`, lookup, or replay consult from any node that reads the same object
   (`reservation_id`, `lease_id`, `(client_operation_id, operation_type, idempotency_key)` triple, or the
   per-`(normalized_licensing_scope, policy_id, window_start)` counter) **MUST** observe the effect of
   that write. Pre-write state for the same object **MUST NOT** be returned after acknowledgment.
2. **Monotonic reads per `correlation_id`.** Within one `correlation_id`, successive reads of the same
   fast-path object **MUST NOT** go backwards in persisted state (no reservation transitioning from
   `committed` back to `reserved` in observed state, no lease transitioning from `released` back to
   `active`). This holds across nodes because all reads traverse the same fast-path store.
3. **Monotonic reads per `client_operation_id`.** Successive admission-producing calls that carry the same
   `client_operation_id` (the business-correlation key used by the idempotent lifecycle of
   `cpt-cf-licensing-service-fr-idempotent-lifecycle`) **MUST** observe a state consistent with all prior
   accepted calls of the same `client_operation_id`; a replay **MUST** match against the persisted
   idempotency record rather than initiate a second admission.
4. **Cache-derived reads are excluded.** Reads served from the entitlement cache, policy cache, or the
   resource-type registry cache are governed by the freshness classification of §4.4 and
   by the caching TTL rules; they are **not** covered by read-your-writes or monotonic-reads guarantees.
   Consumers that require fresh-authoritative evaluation for a lookup **MUST** use the sync-status payload
   (`cpt-cf-licensing-service-fr-sync-status`) to gate on `freshness_state = fresh` rather than assume
   post-write visibility through a cache.
5. **No cross-operation causal chaining without `client_operation_id`.** Two distinct
   `client_operation_id` values are, by construction, independent business operations; the module provides
   no happens-before guarantee between them beyond what the fast-path store itself exposes on shared
   objects (e.g., a shared counter). Consumers that require causal chaining across admissions **MUST**
   reuse `client_operation_id` (subject to the non-reuse rule of
   `cpt-cf-licensing-service-fr-idempotent-lifecycle`) or serialize through their own coordination layer.

Violations of (1)–(3) are implementation bugs, not permitted degraded outcomes; they are not masked by
`decision_state = degraded` or by the `allowed_clock_skew_seconds` boundary tolerance of §4.4 (which
pertains only to `now_at_module` comparisons, not to fast-path object visibility).

### 4.5 External Authority Model

- The external licensing system is upstream-authoritative for legacy-controlled scopes.
- The external authority is **not** part of a distributed transaction with the Licensing Service runtime path.
- Local decisions, reservations, and accounting may temporarily diverge from upstream authority state.
- Reconciliation across local runtime state and external-authority state is eventual, not atomic.
- The module provides one decision boundary, but it does **not** guarantee atomic reserve / capture semantics across
  external systems.

### 4.6 Known Simplifications (p1)

- Quota enforcement may temporarily diverge from entitlement updates.
- Reservation lifecycle in p1 does not include full reconciliation guarantees.
- Idempotency in p1 is key-based and best-effort rather than canonical.
- Policy resolution in p1 does not guarantee byte-identical deterministic ordering across implementations.
- External authority interaction is not transactionally consistent with local runtime admission.
- p1 guarantees only the minimal runtime surface `Check`, `CheckAndReserve`, and `Commit`.

### 4.7 Non-Goals (p1)

- Deterministic cross-node decision replay
- Byte-identical policy resolution
- Strong consistency across the external authority boundary
- Perfect auditability
- Full quota dimension coverage
- Full shadow-comparison automation

### 4.8 Risks

- **Over-centralization**: one module becomes the blast-radius center for licensing and runtime quota decisions.
- **External inconsistency**: upstream authority changes can lag local projections and caches.
- **Reservation leakage**: incomplete settlement or consumer crashes can hold capacity longer than intended.
- **Policy misconfiguration**: bad limits or bad precedence can deny valid work or admit too much work.
- **Performance degradation under load**: centralized decisioning can become a latency bottleneck without warm-path
  discipline.

### 5.1 Canonical API and Domain Model

#### Canonical Licensing API

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-canonical-api`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** provide one canonical API and SDK for licensing and quota decisions across consumer modules.
The minimum p1 runtime surface is `Check`, `CheckAndReserve`, and `Commit`. Later tiers may add
`Release`, `Update`, effective-quota introspection, usage summaries, and lease operations without changing
the core integration model.

- **Rationale**: Modules need one stable integration surface that remains valid across legacy-authoritative and
  native-authoritative deployments.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`

#### Canonical Domain Model

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-canonical-model`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** expose a canonical domain model that includes feature, offering item, entitlement grant, effective
limit, authority, and decision explanation concepts without leaking raw legacy VendorA payloads into consuming modules.

- **Rationale**: Consumers must not couple to authority-specific schemas or legacy business quirks.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-tenant-administrator`

#### Decision Contract

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-decision-contract`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** define a canonical decision contract for decision-producing operations.

- p1 requires `decision_outcome`, `decision_state`, `resolved_authority_mode`, normalized licensing scope
  identifier, primary `reason_code`, `freshness_state`, and `correlation_id`, plus enough metadata to
  separate a business decision from a service error.
- p2 adds `policy_effect`, `contributing_reason_codes`, and basic structured explanation metadata.
- p3 requires the full structured decision contract, canonical-equality behavior, and deterministic payload
  shape across replay, audit, and shadow-comparison paths.
  Serialized field order is not contractually significant unless a later p3 data-contract requirement explicitly
  marks a surface as byte-stable for comparison.

- **Rationale**: A stable decision contract is required so multiple modules and reviewers do not infer different
  semantics from the same response.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`

#### Decision Outcome Semantics

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-decision-outcome-semantics`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** use `decision_outcome = allow` only when the licensing service determines that the request is
permitted by entitlement and applicable limit policy, and **MUST** use `decision_outcome = deny` only when the
licensing service determines that the request is not permitted. Decision outcomes **MUST NOT** encode invalid request,
routing failure, or internal processing failure semantics. `capped`, `downgraded`, `deferred`, and `warned` **MUST NOT**
be
represented as decision outcomes; they **MUST** be represented only through `policy_effect` while preserving
`decision_outcome = allow` or `decision_outcome = deny`. `policy_effect = capped`, `downgraded`, `deferred`, or `warned`
**MUST** be returned only with `decision_outcome = allow`. `decision_outcome = deny` **MUST** use
`policy_effect = none`.

- **Rationale**: Consumers need stable business meaning for allow and deny without overloading outcomes to represent
  transport or processing failures.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`

#### Lookup Response Contract Semantics

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-lookup-contract`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** define canonical response contracts for `GetEffectiveEntitlements` and `GetEffectiveLimits`.

- p1 requires normalized licensing scope identity, resolved authority mode, `correlation_id`, the lookup-specific
  payload, and clear separation between successful responses and service errors.
- p2 adds explicit freshness / degraded metadata and compatibility with the canonical decision contract.
- p3 requires the closed-field-set and deterministic payload-shape guarantees used for replay and comparison.
  Transport-level field order and serialization format are implementation details; the contract is the required
  field set, field meanings, and presence / absence rules stated in this PRD.

- **Rationale**: Read-only lookup operations must not force consumers to guess whether freshness, degraded handling, and
  error semantics differ from decision-producing operations.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`

#### Reason and Error Semantics

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-reason-error-semantics`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** use documented closed-set machine-readable codes for both decision reasons and service errors.

Decision `reason_code` values (covering both the primary `reason_code` field and the
`contributing_reason_codes` list defined in `cpt-cf-licensing-service-fr-decision-contract`) comprise the
closed set:

- **Primary rejection causes (temporal, `R_temporal`)**: `quota_rejected`, `limit_exceeded`,
  `rate_window_exhausted`, `budget_window_exhausted`, `concurrency_saturated`. These always carry
  `retry_after_seconds` per `cpt-cf-licensing-service-fr-retry-after-semantics`.
- **Primary rejection causes (non-temporal, `R_non_temporal`)**: `feature_not_entitled`,
  `offering_item_missing`, `scope_disabled`, `policy_denied`, `authority_data_stale`. These never carry
  `retry_after_seconds`. Note: `invalid_request`, `invalid_correlation_id`, and
  `conflicting_authority_configuration` are **service errors** (see the `error_code` set below), not
  primary rejection `reason_code` values on a canonical decision; they are returned through the service-error
  contract per `cpt-cf-licensing-service-fr-error-distinction` and **MUST NOT** appear as the primary
  `reason_code` of a `decision_outcome = deny` response.
- **Primary admission-outcome reasons (non-rejection)**: `none`, `quota_capped`, `quota_downgraded`,
  `quota_warned`, `quota_deferred`.
- **Contributing-only codes** (emitted exclusively in `contributing_reason_codes`, never as primary):
  `entitlement_pending_activation`, `entitlement_pending_provisioning`, `entitlement_pending_approval`,
  `entitlement_pending_payment`, `quota_downgrade_attempted`. The four `entitlement_pending_*` codes appear
  only alongside either primary `reason_code = quota_deferred` (on entitlement-layer responses carrying
  `policy_effect = deferred`) or primary `reason_code = feature_not_entitled` (on quota-layer rejection
  responses where a reservation or counter advance was suppressed per Rule 2 of
  `cpt-cf-licensing-service-fr-deferred-admission`); `quota_downgrade_attempted` appears only alongside
  `admission_outcome = rejected` when a downgrade path was attempted per
  `cpt-cf-licensing-service-fr-soft-threshold-mechanics`.

Service `error_code` values comprise the versioned vocabulary of the module. The minimal p1 delivery MUST
implement the subset needed by the p1 runtime surface (`Check`, `CheckAndReserve`, `Commit`), authority
routing, and basic administration. Codes introduced for later-tier capabilities such as `Release`, `Update`,
advanced reconciliation, leases, or full shadowing are reserved up front and **MUST NOT** be repurposed. The
cross-tier vocabulary is: `invalid_request`, `invalid_correlation_id`, `invalid_subject_kind`,
`invalid_subject_opaque_id`, `scope_unresolved`, `conflicting_authority_configuration`,
`authority_unavailable`, `internal_failure`, `idempotency_conflict`, `idempotency_record_expired`,
`reservation_expired`, `reservation_not_found`, `policy_config_conflict`, `dimension_set_mismatch`,
`commit_exceeds_reservation`, `overshoot_capacity_exhausted`, `lease_expired`, `lease_already_released`,
`lease_not_found`, `lease_cumulative_lifetime_exceeded`. `reservation_not_found` **MUST** be returned when a `Commit` or
`Release`
references a `reservation_id` that is not present in the fast-path state store and does not match any
idempotency replay record; `lease_not_found` **MUST** be returned analogously for `RenewLease` or
`ReleaseLease` against an unknown `lease_id`. These codes are disjoint from the terminal-state codes
(`reservation_expired`, `lease_expired`, `lease_already_released`) and from `idempotency_record_expired`. The
enumeration above is the reserved cross-tier vocabulary for this PRD. Every listed code **MUST** preserve the
semantics specified here when its owning requirement priority tier is delivered, and PRD-declared codes **MUST NOT**
be dropped or repurposed by downstream documents. The
vocabulary is versioned-extensible rather than strictly closed: additive `error_code` values MAY be
introduced only by a PRD revision to describe additional, non-overlapping failure modes, subject to two
binding rules that make the extension surface safe for SDK retry/abort logic: (i) every additive code
**MUST** be assigned at definition time to exactly one of the closed taxonomy classes enumerated in the
Service-error taxonomy table below (the taxonomy classes themselves **MUST NOT** be extended without a PRD
revision), and (ii) every additive code **MUST** be non-overlapping in scope with every existing code.
When the module implements the behavior an additive code refers to, the code
**MUST** be returned (the additive codes are mandatory-when-applicable, not optional). Additive codes introduced
by this PRD version include `invalid_correlation_id`, `dimension_set_mismatch`, `commit_exceeds_reservation`,
`overshoot_capacity_exhausted`, `policy_config_conflict` (see
`cpt-cf-licensing-service-fr-quota-policy-resolution`), `lease_expired`,
`lease_cumulative_lifetime_exceeded` (see `cpt-cf-licensing-service-fr-concurrency-leases`), `internal_audit_failed` (
see
`cpt-cf-licensing-service-fr-audit-coverage`), `stale_write` (see
`cpt-cf-licensing-service-fr-absolute-stale-write-protection`; returned when an absolute-model write
carries a `version` that is not strictly greater than the persisted per-subject `version` on the same
normalized licensing scope, so the write is rejected without mutating state), and the pair
`invalid_subject_kind` / `invalid_subject_opaque_id` (see `cpt-cf-licensing-service-fr-subject-kind-registry`). The
additive list
above is not reopened to mean that PRD-declared codes may be dropped; the PRD set is the minimum
guaranteed vocabulary for consumers.

Policy-admission and registry-admission validation failures are configuration defects, not runtime caller-input
defects. Therefore quota-policy admission, resource-type declaration admission, subject-kind registration,
and emergency-override upsert **MUST** use `error_code = policy_config_conflict` for omitted required fields,
illegal values, unsupported enum members, range violations, and semantic collisions. `error_code = invalid_request`
is reserved for malformed or semantically invalid runtime request payloads on decision, reservation, settlement,
lease, and lookup paths. When one configuration submission contains multiple validation defects, the module MAY
return any one applicable `policy_config_conflict`, but **MUST** reject the submission atomically.

`dimension_set_mismatch` **MUST** be returned when a `Commit` request names a set of resource-dimension keys that
differs from the persisted `reserved_quantity` key set of the referenced reservation. This key-set equality check
occurs before per-dimension magnitude validation, overshoot validation, or capacity evaluation; when both a
dimension-key mismatch and a magnitude excess are present, `dimension_set_mismatch` wins.

Every additive `error_code` introduced by a future PRD revision **MUST** be classified into exactly one

### 5.2 Federated Authority Routing

#### Normalized Licensing Scope

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-normalized-scope`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** derive a `Normalized Licensing Scope` for every decision-producing operation and every effective
entitlement or effective-limit lookup. The normalized scope **MUST** use the canonical tuple
`(tenant_scope, subject_scope, product_family, offering_item_or_feature, deployment_environment, usage_dimension)`.
The same normalized scope tuple **MUST** be used for authority routing, cache identity, audit records, and shadow
comparison for the same logical request.

- **Rationale**: Routing, caching, auditing, and migration comparison are unsafe if different teams normalize the same
  request differently.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`,
  `cpt-cf-licensing-service-actor-migration-engineer`

#### Subject Kind Registry

- [ ] `p3` - **ID**: `cpt-cf-licensing-service-fr-subject-kind-registry`

- **Priority**: p3
- **Strictness**: RELAXED

The system **MUST** govern concrete per-subject identities inside `subject_scope` through a platform-wide
**Subject Kind registry**, structurally analogous to the resource-type registry
(`cpt-cf-licensing-service-fr-resource-types`). Every concrete `subject_scope` value in a normalized licensing
scope tuple (and in `scope_pattern.subject_scope` when non-wildcard) **MUST** conform to the canonical encoding

```
subject_scope ::= "_scope_root_" | "_none_" | <subject_kind> ":" <opaque_id>
```

where `<subject_kind>` is a registered identifier matching `^[a-z][a-z0-9_]*(\.[a-z][a-z0-9_]*)*$` with length
`3..=128`, and `<opaque_id>` is any non-empty UTF-8 byte sequence drawn from `^[A-Za-z0-9._:\-]+$` with length
`1..=256` (opaque to Licensing Service; its structure is owned by the registering module). Canonical equality
over `subject_scope` is **byte-wise over the full encoded string**; no Unicode normalization, no case-folding,
no trimming.

below. Submissions that omit a REQUIRED field or that carry an illegal value are rejected with
`error_code = policy_config_conflict`; conflicting re-registration attempts are rejected with
`error_code = policy_config_conflict`.

| Field                  | Type             | Presence                                | Semantics                                                                                                                                                                                                                                  |
|------------------------|------------------|-----------------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `subject_kind`         | `String`         | REQUIRED                                | Matches `^[a-z][a-z0-9_]*(\.[a-z][a-z0-9_]*)*$`, length `3..=128`. Case-insensitive collisions (e.g. `ams` vs `AMS`) are rejected at registration with `error_code = policy_config_conflict`.                                              |
| `owning_module_id`     | `String`         | REQUIRED                                | Identifier of the module that owns the semantic meaning of `<opaque_id>` values in this kind. Used for deregistration authorization and for operator attribution.                                                                          |
| `semver`               | `String`         | REQUIRED                                | SemVer 2.0.0 version of this registration. Additive-only evolution is non-breaking; changing `opaque_id_pattern` or `opaque_id_max_length` in a way that rejects previously accepted values is breaking and requires a major-version bump. |
| `opaque_id_pattern`    | `String`         | OPTIONAL, default `^[A-Za-z0-9._:\-]+$` | ECMAScript-style regex that every `<opaque_id>` of this kind must satisfy. Narrowing the default pattern is permitted; widening it requires a major-version bump.                                                                          |
| `opaque_id_max_length` | `uint32`         | OPTIONAL, default `256`                 | Upper length bound for `<opaque_id>` strings of this kind.                                                                                                                                                                                 |
| `retired_at`           | `RFC3339_UTC_ms` | OPTIONAL                                | When present and `<= now_at_evaluation` under §4.4 Clock authority, new policies and new request-side writes whose `subject_scope` uses this kind **MUST** be rejected. Existing records remain readable for audit and reconciliation.     |

Re-submitting an identical `(subject_kind, semver)` pair with byte-identical field values is idempotent (no-op).
Two submissions sharing the same `subject_kind` but differing in `owning_module_id`, `opaque_id_pattern`, or
`opaque_id_max_length` are a conflict and **MUST** be rejected.

##### Runtime admission rules

1. Every caller-supplied `subject_scope` value on a state-changing or state-producing request
   (`CheckAndReserve`, `Commit`, `Release`, `Update`, `AcquireLease`, `RenewLease`, `ReleaseLease`, policy
   admission) **MUST** be either `_scope_root_`, `_none_` (where semantically permitted), or match the
   canonical `<subject_kind>:<opaque_id>` encoding with a `<subject_kind>` currently present in the registry
   and a non-retired registration. Non-conforming values are rejected under the following closed mapping:
   (a) `<subject_kind>` segment fails the syntax regex, is not present in the registry, or points at a
   registration whose `retired_at <= now_at_evaluation` — `error_code = invalid_subject_kind`; (b)
   `<subject_kind>` is valid but `<opaque_id>` fails `opaque_id_pattern` or exceeds `opaque_id_max_length` for
   that kind — `error_code = invalid_subject_opaque_id`; (c) the overall `subject_scope` value fails the
   high-level grammar check (missing separator, empty segment, illegal character outside both segments, or any
   other malformation that cannot be attributed to exactly one segment) — `error_code = invalid_request`.
   Both additive codes belong to the `client_error` taxonomy class per
   `cpt-cf-licensing-service-fr-reason-error-semantics` and **MUST NOT** carry `retry_after_seconds`.
2. Read-only preview and lookup operations (`Check`, `CheckFeatureEntitlement`, `GetEffectiveEntitlements`,
   `GetEffectiveLimits`, `GetLicensingDecision`, `GetEffectiveQuota`, `GetUsageSummary`) **MUST** apply the
   same validation; an invalid encoding is a request defect, not a decision.
3. Shadow comparison treats `subject_scope` as a single canonical string and **MUST NOT** parse or compare
   `<subject_kind>` and `<opaque_id>` separately. Two sides that differ in either part produce a
   `scope-mismatch`-class signal at the canonical-equality layer.
4. Implementations **MAY** cache registry lookups, but caching strategy **MUST NOT** alter ingress validation,
   canonical equality, or wildcard-matching semantics.

##### Policy-side wildcarding

On the policy side, `scope_pattern.subject_scope` is either the reserved wildcard `_any_` (matches every
request-side `subject_scope` value) or a concrete encoded value (`_scope_root_`, `_none_`, or
`<subject_kind>:<opaque_id>` drawn from the same encoding rules). "Wildcard over a subject kind only"
(e.g. "any opaque_id of kind `ams`") is **NOT** supported in p1; policies that need such semantics must be
expressed through one or more explicit patterns. Extending the pattern grammar to support kind-scoped wildcards
is a non-breaking future extension provided the canonical serialization rules of `scope_pattern_sort_key`
(Glossary) are extended in lockstep.

##### Non-goal

The Subject Kind registry does **not** expand the closed scope-level enumeration defined in Glossary
*Scope Level*. Scope axes and subject kinds serve different purposes: axes define which `scope_level` label
attaches to admission decisions, whereas subject kinds define the grammar of concrete per-subject identifiers
within `subject_scope`. A consumer module that needs a new **scope axis** (not a new subject identity) must
still pursue a PRD update per the extension rule on *Scope Level*.

##### Interaction with hierarchical-scope candidate-tuple expansion

A request may carry zero or more per-subject identities through request context fields whose values encode as
`subject_scope = <kind>:<id>`. Candidate-tuple expansion is purely mechanical: for each supplied subject identity,
the resolver contributes one candidate tuple with that exact encoded `subject_scope`, and it always contributes the
scope-root tuple using `subject_scope = _scope_root_`. The resolver **MUST NOT** synthesize identities of another
kind from an existing `subject_scope` value. For example, `subject_scope = credential:<id>` contributes
`credential:<id>` only when `credential` is registered and supplied by the request; it does not imply `user:<id>` or
`project:<id>`. Policies that need to match a domain-specific subject kind **MUST** declare either the literal
`scope_pattern.subject_scope = <kind>:<id>` or the wildcard `_any_` (pending the kind-scoped wildcard extension
noted under *Policy-side wildcarding* above).

- **Rationale**: Consumer modules consistently need to carry per-subject identities whose semantic type is not one of
  the generic platform scope levels (credentials, installations, agents, devices, namespaces, pods, buckets, accounts,
  projects-in-third-party-systems). Without a registry these identities either get smashed into existing axes (losing
  type safety) or force repeated PRD updates to expand the scope-level closed set. A registered subject-kind prefix gives
  consumers a universal, type-tagged encoding within `subject_scope` without widening the admission or audit contract,
  and matches established industry patterns (AWS ARN, Kubernetes GVK, OpenTelemetry resource attributes) for extensible
  typed identifiers under a fixed canonical structure.
- **Actors**: `cpt-cf-licensing-service-actor-service-integrator`, `cpt-cf-licensing-service-actor-platform-operator`,
  `cpt-cf-licensing-service-actor-control-plane`

#### Explicit Authority Mode per Licensing Scope

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-authority-routing`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** resolve an explicit authority mode for each licensing scope and **MUST** support at least

deterministic for the same normalized licensing scope input. If no authority mode can be resolved, the system **MUST**

**MUST NOT** depend on caller-local normalization or undocumented defaults.

- **Rationale**: Hybrid deployments require deterministic routing and migration behavior rather than implicit fallback.
- **Actors**: `cpt-cf-licensing-service-actor-platform-operator`, `cpt-cf-licensing-service-actor-consumer-module`,

- **Priority**: p1

The system **MUST NOT** allow the same licensing scope to be simultaneously authoritative in both legacy and native

- **Rationale**: A scope must have exactly one business source of truth to avoid conflicting entitlement decisions.

#### Disabled Mode Semantics

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-disabled-mode`

- **Priority**: p1
- **Strictness**: REQUIRED

For any decision-producing operation whose resolved authority mode is `disabled`, the system **MUST** return a

- **Rationale**: `disabled` must have one stable product meaning so callers do not interpret it as either business

- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`

#### No Cross-Authority Failover

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-no-cross-authority-failover`

- **Priority**: p1
- **Strictness**: REQUIRED

After resolving the authority mode for a licensing scope, the system **MUST NOT** switch the request to a different
authority because the resolved authority is stale, slow, or unavailable. Cross-authority substitution is forbidden;
the system may return only a service error or a policy-permitted degraded decision derived from the resolved authority
context. The resolved authority mode for a given normalized licensing scope **MUST** remain immutable for the
lifetime of a single request evaluation; re-resolving within the same request is forbidden. "Request evaluation"
here denotes a single API invocation, not a business operation spanning multiple `client_operation_id`-correlated
calls.

- **Rationale**: Silent cross-authority fallback would create hidden dual-truth behavior and unsafe migration outcomes.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`

#### Freshness and Degraded Decision Semantics

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-freshness-semantics`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** classify authority data freshness using the closed set `fresh`, `stale_within_policy`, and
`stale_beyond_policy`. A decision with `decision_state = authoritative` **MUST** be based only on `fresh` authority
data. A decision with `decision_state = degraded` **MUST** be returned only when product policy explicitly permits use
of `stale_within_policy` data or another non-authoritative evaluation path. `stale_beyond_policy` data **MUST NOT** be
represented as authoritative. Consolidating the closed mapping implied by this requirement and by
`cpt-cf-licensing-service-fr-degraded-policy`: `fresh` data **MUST** produce `decision_state = authoritative`
(there is no permitted path by which a freshness-classified-`fresh` read yields `decision_state = degraded` in p1;
introducing such a path requires a PRD update); `stale_within_policy` data **MUST** produce either
`decision_state = degraded` (when product policy permits degraded evaluation for the operation class and scope)
or a service error; `stale_beyond_policy` data **MUST** produce either `decision_state = degraded` (only when
product policy explicitly marks the scope as degraded-eligible for that operation class) or a service error.

- **Rationale**: Freshness semantics must be explicit so consumers can distinguish valid authoritative outcomes from
  tolerated degraded outcomes.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`,
  `cpt-cf-licensing-service-actor-migration-engineer`

#### Degraded Eligibility by Operation Class

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-degraded-policy`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** apply degraded-decision policy by operation class. The operation classes that **MAY** return
`decision_state = degraded` are limited to the following closed set: `CheckFeatureEntitlement`,
`GetEffectiveEntitlements`, `GetEffectiveLimits`, `GetLicensingDecision`, `Check` (quota preview),
`GetEffectiveQuota`, and `GetUsageSummary`. Degraded decisions **MUST** be returned only when product policy
explicitly marks the normalized licensing scope as degraded-eligible for that operation class. Degraded eligibility
and freshness thresholds **MUST** be defined for each product family before implementation of that product family
and **MUST** be evaluated using the same normalized licensing scope and operation class as the live request. The
"freshness thresholds" referenced here are exactly the two boundary values separating the closed set
`{fresh, stale_within_policy, stale_beyond_policy}` defined in `cpt-cf-licensing-service-fr-freshness-semantics`:
`fresh_threshold` (inclusive upper bound of `fresh`: staleness `s` satisfies `s <= fresh_threshold`) and
`stale_policy_threshold` (inclusive upper bound of `stale_within_policy`: staleness satisfies
`fresh_threshold < s <= stale_policy_threshold`). Both are expressed in integer seconds. Module-wide fallback defaults
for these thresholds **MAY** exist but
**MUST NOT** be used to elevate a product family to degraded-eligible in the absence of explicit per-product-family
configuration. Any fallback defaults are startup configuration conveniences only and **MUST** satisfy
`0 <= fresh_threshold <= stale_policy_threshold`; policy admission for a degraded-eligible product family **MUST**
reject missing thresholds rather than silently inheriting degraded eligibility.
State-changing operations including `CheckAndReserve`, `Commit`, `Release`, entitlement-management operations,
and authority-configuration operations **MUST NOT** return degraded success results; when fresh authoritative
evaluation is unavailable they **MUST** return service errors. Until degraded eligibility and freshness thresholds
have been configured for a product family, the module **MUST NOT** return `decision_state = degraded` for that
product family and **MUST** return service errors when fresh authoritative evaluation is unavailable.

- **Rationale**: Read-only decision paths and state-changing operations need different safety rules under stale or
  unavailable authority data.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`,
  `cpt-cf-licensing-service-actor-native-licensing-store`

#### Authority-Aware Explanation

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-explanation`

- **Priority**: p2
- **Strictness**: REQUIRED

The system **MUST** return explanation metadata, and the required depth follows the p1/p2/p3 priority model.

- p1: every decision MUST expose `decision_outcome`, primary `reason_code`, `correlation_id`, and enough
  metadata to distinguish entitlement-driven denial from quota-driven denial.
- p2: decisions MUST additionally surface `policy_effect`, resolved authority identity, freshness context,
  and a basic structured explanation suitable for operator diagnostics.
- p3: decisions MUST carry the full `decision_explanation` graph, including limiting scopes, policy
  identifiers, contributing and masked policies, effective limits, and deterministic field-shape rules.
  The PRD owns the required fields and presence rules named by the relevant interface and feature requirements.
  Serialization format, internal builder structure, and transport-specific field ordering are implementation
  details and are not p1 exit criteria unless an FR explicitly marks a byte-stable comparison surface.

- **Rationale**: Operators and integrators need to understand whether a denial or restriction came from legacy
  entitlement, native entitlement, or runtime quota enforcement; and need a machine-comparable explanation payload so
  audit replay, shadow analysis, and operator tooling agree on what the module emitted.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`,
  `cpt-cf-licensing-service-actor-tenant-administrator`

#### Error and Degraded Decision Distinction

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-error-distinction`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** distinguish service errors from decision objects. A service error **MUST** be returned when no
semantically valid licensing decision can be produced. A degraded decision **MUST** still conform to the canonical
decision contract, **MUST** carry `decision_state = degraded`, and **MUST** include the reason that non-authoritative
evaluation was used.

- **Rationale**: Consumers and reviewers must not confuse degraded business decisions with failures to produce any
  decision at all.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`

### 5.5 Internal Ownership and Decision Orchestration

#### Internal Layer Ownership

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-internal-ownership`

- **Priority**: p2
- **Strictness**: RELAXED

The system **MUST** maintain distinct internal ownership between the entitlement layer and the quota layer such that
the quota layer does not own or redefine commercial licensing truth (entitlement existence, feature enablement,
effective licensed limit, or commercially authoritative consumed quantity) for any scope that is authoritative in
the entitlement layer. Concretely, the entitlement layer owns entitlement-grant existence and lifecycle, feature
enablement, effective licensed limit, commercially authoritative consumed quantity, and authority-emission freshness
inputs; the quota layer owns quota policies, reservations, leases, per-window counters, replay records, soft quota
modifiers, and usage-summary / remaining-capacity views derived from quota enforcement. Quota-layer-owned quantities
**MUST NOT** mutate entitlement truth unless a product-specific extension to this PRD explicitly reclassifies that
usage class before implementation for that product family.

- **Rationale**: Unified module does not mean merged ownership; internal layers must preserve clear authority
  boundaries.

The system **MUST** support canonical feature semantics metadata. At minimum this metadata model **MUST** support
`activity_type` (`persisted` or `transient`), `quantity_model` (`counted`, `incremental`, or `absolute`), canonical

to distinguish online replay safety from historical retention.

- **Rationale**: Some runtime behaviors depend on whether a limit governs durable state or bounded activity and on how
  quantity changes must be interpreted over time.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-service-integrator`

### 5.6 Quota Policy Management and Resolution

#### Policy Lifecycle Administration

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-quota-policy-lifecycle`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** allow authorized control-plane clients to list, create, update, disable, and expire quota policies
for platform, region, tenant, project, and user scopes. Policies **MAY** further narrow applicability through
`scope_pattern.subject_scope` for registered domain-specific subject kinds and through `product_family`,
`offering_item_or_feature`, `deployment_environment`, and `usage_dimension` components of the normalized licensing
scope.

Every quota policy record carries a lifecycle field `policy_state` drawn from the closed set
`{ active, disabled, expired }`, orthogonal to the `category` enum defined in
`cpt-cf-licensing-service-fr-quota-policy-resolution`. Semantics:

- `active`: the record participates in `P_applicable` per `cpt-cf-licensing-service-fr-quota-policy-resolution`
  and in `masked_policies` audit when masked by a higher-precedence record.
- `disabled`: the record is administratively suspended. The resolver **MUST** exclude `disabled` records from
  `P_applicable` entirely, **MUST NOT** surface them in `decision_explanation.masked_policies`, and **MUST NOT**
  use them for precedence filtering (so a `disabled` `scoped` record does not mask a `default` record). Disabling
  is the operator-facing mechanism for temporarily lifting a policy without deletion; re-enabling is an
  administrative state transition that **MUST** be auditable. From p2 onward, when
  `cpt-cf-licensing-service-fr-audit-coverage` is in scope, the transition **MUST** be recorded through that
  FR's canonical audit path; in p1 the transition is visible through the policy-lifecycle API and operator
  telemetry without a structured audit record.
- `expired`: the record has passed its configured expiry instant. Semantically equivalent to `disabled` for
  `P_applicable` purposes but irreversible through the admin API.

Lifecycle transitions `active ↔ disabled` and `active → expired`, `disabled → expired` **MUST** be auditable
under the same staged rule as above: from p2 onward, when `cpt-cf-licensing-service-fr-audit-coverage` is in
scope, each transition **MUST** produce a canonical audit record through that FR; in p1 the transition is
observable through the policy-lifecycle API and operator telemetry without a structured audit record.

Quota-policy policy-admission validation **MUST** enforce the calendar-window invariants that govern
runtime window attribution: `period ∈ { hourly, daily, weekly, monthly, billing_cycle }`, and for
`period = billing_cycle` the required per-policy `billing_anchor_day: uint8` field **MUST** be in the
closed integer range `[1, 28]` (values `29..31` **MUST** be rejected at policy admission with
`error_code = policy_config_conflict` to avoid month-length ambiguity; concrete boundary-alignment rules
for each `period` value follow Glossary *Current Quota Window*). Sliding-window and
token-bucket rate-limit shapes are out of p1 scope; declaring any `period` value outside the closed set
above **MUST** also be rejected with `error_code = policy_config_conflict`.

- **Rationale**: Platform and tenant governance require controlled lifecycle management of quota rules instead of ad hoc
  runtime configuration.
- **Actors**: `cpt-cf-licensing-service-actor-platform-operator`, `cpt-cf-licensing-service-actor-tenant-administrator`,
  `cpt-cf-licensing-service-actor-control-plane`

#### Deterministic Inheritance and Override Resolution

- [ ] `p3` - **ID**: `cpt-cf-licensing-service-fr-quota-policy-resolution`

- **Priority**: p3
- **Strictness**: RELAXED

The system **MUST** resolve effective quota policy according to a priority-tiered model.

- p1: support flat/default policies and a simple scoped override model sufficient for `Check`,
  `CheckAndReserve`, and `Commit`. p1 does not require full candidate-tuple expansion, byte-identical
  serialization of `P_applicable`, or total-order equivalence across independent implementations.
- p2: support scoped policies, explicit conflict handling, and a simple precedence model appropriate for
  operational administration.
- p3: support the full normalized-scope hierarchy, deterministic expansion, precedence, and replay / audit
  determinism of the effective policy set.
  Across all delivered tiers, resolution **MUST** be deterministic for identical inputs: the same normalized
  licensing scope, requested dimensions, and policy catalog **MUST** produce the same `P_applicable`, the same
  precedence / masking outcome, and the same tie-break result. Internal snapshot identifiers, storage version
  tokens, and serialization format are implementation details.

- **Rationale**: Consumer modules and operators need predictable enforcement and a stable explanation for every quota
  decision; two independent implementations must construct the same `P_applicable` for identical inputs so that
  admission, shadow comparison, and audit replay agree.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`,
  `cpt-cf-licensing-service-actor-control-plane`

#### Administrative Audit Coverage

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-audit-coverage`

- **Priority**: p2
- **Strictness**: REQUIRED

The system **MUST** produce a durable audit record for every administrative state transition that changes
licensing, quota, routing, or registry truth visible to admission. The covered closed set of **auditable
administrative actions** is:

1. **Quota policy lifecycle** (`cpt-cf-licensing-service-fr-quota-policy-lifecycle`): creation, update,
   `policy_state` transition (`active ↔ disabled`, `active → expired`, `disabled → expired`), deletion, and
   any change to `scope_pattern`, `effective_limit`, `warn_threshold_percent`, `cap_enabled`,
   `downgrade_target_policy_id`, `rejection_reason_code`, `counted_unit`, `period`, `billing_anchor_day`,
   `default_lease_ttl_seconds`, `max_cumulative_lease_lifetime_seconds`, `allow_overshoot_bound_percent`, or
   `degradation_policy`.
2. **Authority-routing configuration** (`cpt-cf-licensing-service-fr-authority-routing`): creation, update,
   or deletion of any authority-rule record; every change of `resolved_authority_mode` for a normalized
   licensing scope (including transitions to or from `disabled` and `shadow`).
3. **Emergency override lifecycle** (`cpt-cf-licensing-service-fr-safe-degradation`): creation,
   update, explicit revocation, and natural expiry (`now_at_evaluation >= expires_at`) of every
   `emergency_override` record.
4. **Subject Kind registry mutations** (`cpt-cf-licensing-service-fr-subject-kind-registry`): registration,
   SemVer-bumped update, and retirement (`retired_at` set) of any subject-kind record.
5. **Resource type registry mutations** (`cpt-cf-licensing-service-fr-resource-types`): registration,
   SemVer-bumped update, and retirement of any resource-type record.
6. **Native entitlement mutations for native-authoritative scopes**
   (`cpt-cf-licensing-service-fr-native-authority`): creation, update, and revocation of entitlement grants;
   transitions of the entitlement-grant lifecycle state closed set `{ active, pending_activation,
   pending_provisioning, pending_approval, pending_payment, expired, revoked, absent }` that are driven by
   the module's own admin API (authority-driven transitions on legacy-authoritative scopes remain audited
   by the source authority and are out of scope for this FR).

Every audit record **MUST** carry, at minimum:

- `correlation_id` drawn from the admin request (generated per Glossary *Correlation ID* when the admin API
  call omits it);
- `actor_id` identifying the acting principal, one of `cpt-cf-licensing-service-actor-platform-operator`,
  `cpt-cf-licensing-service-actor-tenant-administrator`, `cpt-cf-licensing-service-actor-control-plane`,
  `cpt-cf-licensing-service-actor-migration-engineer`, `cpt-cf-licensing-service-actor-service-integrator`,
  or a platform-authenticated system identity (reconciliation-worker-driven mutations are recorded under
  `cpt-cf-licensing-service-actor-reconciliation-worker`);
- `action` drawn from the closed set enumerated above (e.g. `policy_state_disabled`,
  `emergency_override_created`, `subject_kind_retired`);
- `target_id` — the stable identifier of the mutated record (`policy_id`, `subject_kind`,
  `resource_type_id`, scope tuple for authority rules, entitlement-grant identifier);
- `before` and `after` snapshots of the mutated record under *Canonical Serialization of `P_applicable`*
  rules (Glossary) for byte-stable diff reconstruction; absent fields on the pre-image (`create`) or
  post-image (`delete`) are encoded as explicit `null`;
- `occurred_at: RFC3339_UTC_ms` — wall-clock instant of the mutation, sourced per §4.4 Clock authority rule
  (1) from the durable accounting store's commit-time timestamp.

Audit records **MUST** be persisted to the same durable audit path governed by
`cpt-cf-licensing-service-nfr-auditability` and retained under its retention threshold. Failure to produce
the audit record **MUST** fail the admin mutation with `error_code = internal_audit_failed`; partial
success (mutation persisted but audit record absent) is forbidden. The mutation and its audit record
**MUST** commit atomically — either both are durable after commit or neither is — so that replay of the
durable accounting store's log always reconstructs both sides in agreement.

Read-only operations (`Check`, `CheckFeatureEntitlement`, `GetEffectiveEntitlements`, `GetEffectiveLimits`,
`GetLicensingDecision`, `GetEffectiveQuota`, `GetUsageSummary`, authority-status introspection, shadow
comparison retrieval) are **not** administrative actions and are **not** covered by this FR; their
observability is governed by `cpt-cf-licensing-service-nfr-observability` and
`cpt-cf-licensing-service-nfr-quota-observability`.

- **Rationale**: Admission truth is the product of policy catalog, authority routing, emergency overrides, and
  the two registries (subject-kind and resource-type); an admission outcome can only be explained post-factum
  when every mutation to those four sources is reconstructible with actor, before/after state, and timing.
  A single closed set of auditable actions — bound to the FRs that own them and the atomicity rule above —
  prevents drift between "what happened" and "what the audit log claims".
- **Actors**: `cpt-cf-licensing-service-actor-platform-operator`, `cpt-cf-licensing-service-actor-control-plane`,
  `cpt-cf-licensing-service-actor-tenant-administrator`

#### Hierarchical Scope Evaluation

- [ ] `p3` - **ID**: `cpt-cf-licensing-service-fr-hierarchical-scope-evaluation`

- **Priority**: p3
- **Strictness**: RELAXED

The system **MUST** evaluate all applicable platform, region, tenant, project, and user scopes for a request and
**MUST** compute effective remaining quota from the most restrictive applicable hard constraint while preserving
applicable soft-limit signals separately. Domain-specific quota constraints (for example per credential, per
installation, per device, per target class, per storage class, or per provider SKU) **MUST** participate through
`scope_pattern` matching over registered `subject_scope`, `product_family`, `offering_item_or_feature`, and
`usage_dimension` values rather than through additional global scope levels.

- **Rationale**: Hierarchical quotas are only safe when all applicable scopes participate in a deterministic evaluation
  model.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`,
  `cpt-cf-licensing-service-actor-control-plane`

#### Domain-Specific Quota Dimensions

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-domain-specific-dimensions`

- **Priority**: p2
- **Strictness**: REQUIRED

The system **MUST** support product-specific quota semantics without adding product-specific values to the global
`Scope Level` enum. A product-specific quota dimension **MUST** be represented by one or more of the following
canonical mechanisms:

1. A registered `subject_kind` inside `subject_scope` when the dimension identifies the caller, owner, installation,
   device, credential, account, namespace, or another per-subject identity.
2. A `product_family` or `offering_item_or_feature` value when the dimension selects the licensed product, commercial
   offering, feature, SKU, capability, or provider-specific target class being governed.
3. A registered `usage_dimension` backed by `cpt-cf-licensing-service-fr-resource-types` when the dimension measures
   consumed capacity, such as requests, bytes, tokens, cost units, seats, jobs, storage items, or another integer
   quantity.

Quota-policy admission **MUST** reject a product-specific quota policy that attempts to encode a new domain concept by
using an out-of-set `scope_level`. Product-specific dimensions participate in `P_applicable` only through normal
`scope_pattern` matching over the normalized licensing scope and through `governs(p, d)` for registered resource
dimensions. They do **not** create extra scope-narrowness levels; when two binding policies tie on normalized shortfall
and canonical scope level, the deterministic tie-break remains `policy_id` lexicographic order.

- **Rationale**: Licensing Service must remain a reusable platform module. Domain concepts belong in registries and
  normalized-scope values owned by the consuming product, not in the global scope hierarchy.
- **Actors**: `cpt-cf-licensing-service-actor-service-integrator`, `cpt-cf-licensing-service-actor-platform-operator`,
  `cpt-cf-licensing-service-actor-control-plane`

### 5.7 Runtime Decisioning and Enforcement Semantics

#### Quota Class Coverage

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-quota-class-coverage`

- **Priority**: p2
- **Strictness**: RELAXED

The system **MUST** support rate quotas, budget or consumption quotas, and concurrency quotas as distinct first-class
policy kinds and **MUST** allow multiple quota kinds to be applied to the same subject and action context.

- **Rationale**: Different quota classes have different semantics and cannot be safely collapsed into one generic
  counter model.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-service-integrator`

#### Multi-Dimensional Quota Evaluation

- [ ] `p3` - **ID**: `cpt-cf-licensing-service-fr-multi-dimensional-decisioning`

- **Priority**: p3
- **Strictness**: RELAXED

The system **MUST** evaluate multiple resource dimensions in the same decision, including requests, tokens, storage,
cost units, concurrency, and custom resource types, and **MUST** reject the decision when any applicable hard constraint
is violated even if other dimensions remain available.

- **Rationale**: Real workloads consume several constrained resources at once and cannot be governed correctly by a
  single counter or a single scope.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`

#### Reservation-Based Check and Reserve

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-check-and-reserve`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** support a low-latency `CheckAndReserve` operation that accepts a `client_operation_id`, an
`idempotency_key`, execution context, estimated resource dimensions, and an optional `requested_ttl_seconds` and,
when permitted, returns a stable reservation identifier together with reservation state and `expires_at`. When the
caller supplies `requested_ttl_seconds`, the module **MUST** silently clamp it by applying
`effective_ttl = min(requested_ttl_seconds, default_reservation_ttl_seconds)`; supplying a value larger than
`default_reservation_ttl_seconds` is **not** a client error and **MUST NOT** yield
`error_code = invalid_request`. A value of `0` (or any non-positive integer after decoding) **MUST** be rejected
with `error_code = invalid_request`. Both `default_reservation_ttl_seconds` and `requested_ttl_seconds` are
integer seconds (`uint32`). `default_reservation_ttl_seconds` is a **required**
policy field with valid range `[1, max_reservation_ttl_seconds]`; quota policies that omit it or declare a
value of `0` (or any non-positive value after decoding) **MUST** be rejected at policy admission time with
`error_code = policy_config_conflict`. The module-wide hard cap
`max_reservation_ttl_seconds` bounds `default_reservation_ttl_seconds` from above and is defined in the Glossary as
a module-wide startup-configurable constant (p1 default `900`, allowed range `[1, 86_400]`). On successful reservation
acceptance, the module persists `accepted_at`, derives `expires_at = accepted_at + effective_ttl`, and persists
`reserved_quantity[d]` as the admitted magnitude actually bound against capacity for each requested dimension `d`:
on `admitted_with_cap` it equals the returned `admitted_magnitude[d]`; on every other successful reserve path it
equals the accepted `requested_quantity[d]`. `Commit` whose dimension-key set differs from the persisted
`reserved_quantity` key set **MUST** be rejected with `error_code = dimension_set_mismatch` before any per-dimension
magnitude or overshoot checks. `Commit`
and `Release` against a reservation whose persisted or boundary-classified state is `expired` **MUST** be rejected
with `error_code = reservation_expired`; p1 provides no reservation-renewal API. Repeating the same semantic request
with the same `idempotency_key` **MUST** return the same logical outcome and **MUST NOT** create an additional
reservation. Preview, reserve, commit, and release calls that correspond to the same real-world operation **MUST**
be correlated by a shared `client_operation_id` even when each call carries its own distinct `idempotency_key`.

- **Rationale**: Expensive and long-running operations require a pre-execution guard that prevents obvious overspend
  before work begins.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`

#### Reservation Lifecycle Guarantees

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-reservation-lifecycle`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** support reservation lifecycle semantics, but delivery is staged.

- p1: reservations use the simplified closed set `{ reserved, committed, expired }`. `Commit` settles a live
  reservation. Expiry returns held capacity. Double-commit, commit against unknown reservations, and commit
  against expired reservations are rejected.
- p2: the `released` state is added for caller-initiated early return of unused capacity, and reconciliation
  is introduced to recover leaked reservations. ADR-0003 becomes operationally applicable from p2 onward.
- p3: the `reconciled` audit flag and full post-terminal audit semantics are required. `reconciled` is
  orthogonal to lifecycle state: setting it **MUST NOT** change lifecycle state or capacity arithmetic, and it
  **MUST NOT** create a fifth lifecycle state. Overshoot handling, when delivered, is evaluated on `Commit`
  against the reservation originally admitted and **MUST NOT** create a new lifecycle state.
  p1 deliberately does not require the full production-grade distributed recovery state machine.

- **Rationale**: Reservation-based enforcement is only reliable when lifecycle transitions are explicit, bounded, and
  safe under retries.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-reconciliation-worker`

#### Release Outcome Semantics

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-release-outcome`

- **Priority**: p2
- **Strictness**: REQUIRED

The system **MUST** accept an optional caller-supplied `release_outcome` field on every `Release` and
`ReleaseLease` request, drawn from the closed set `{ released, failed, cancelled }`:

- `released`: the guarded operation completed successfully and the caller is returning any unused capacity.
  This is the default when `release_outcome` is omitted by the caller; for idempotency canonical-equality
  purposes an omitted `release_outcome` on a `Release` or `ReleaseLease` request **MUST** be normalized to
  the literal `released` before comparison (see item 4 below), so an initial call that omits the field and
  a replay that explicitly supplies `release_outcome = released` (or vice versa) are canonical-equality
  equivalents and **MUST NOT** produce `idempotency_conflict`.
- `failed`: the guarded operation started but did not complete successfully (consumer-side error, upstream
  failure, or partial execution). Capacity is returned to the scope identically to `released`; the distinction
  is recorded for audit and reconciliation classification.
- `cancelled`: the guarded operation was aborted before meaningful execution began (consumer-initiated
  cancel, timeout before start, or policy-directed abort after reservation).

`release_outcome` **MUST NOT** alter the reservation or lease lifecycle transition. Every accepted `Release`
transitions the reservation from `reserved` to `released`, and every accepted `ReleaseLease` transitions the
lease from `active` to `released`, regardless of the outcome label; the closed lifecycle state set of
`cpt-cf-licensing-service-fr-reservation-lifecycle` and of `cpt-cf-licensing-service-fr-concurrency-leases`
is unchanged. `release_outcome` **MUST NOT** alter capacity arithmetic: the full unused reserved magnitude
(or the full lease slot) is returned to the applicable scope in every case. The field is a caller-side
semantic annotation that **MUST** be:

1. Persisted on the terminal reservation or lease record for audit and reconciliation introspection.
2. Echoed back on the response payload as `release_outcome` so that idempotent replays return the same value
   and shadow-comparison canonical equality can detect implementation divergence.
3. Exposed through `cpt-cf-licensing-service-fr-usage-summary` as a breakdown within the `released` lifecycle
   bucket. To prevent the sub-count `released` from colliding with its parent `released` lifecycle bucket in
   flattened telemetry surfaces (tags, column names, single-level metric keys), the three sub-counts **MUST**
   be emitted under distinct, namespaced keys `released.by_outcome.released`,
   `released.by_outcome.failed`, and `released.by_outcome.cancelled` (summing to the total `released` count);
   a single top-level `released` metric that shadows the lifecycle bucket **MUST NOT** be emitted as the
   name of any sub-count. Operators can thereby observe the ratio of
   successful-versus-failed-versus-cancelled releases without parsing free-form explanation text. The
   breakdown **MUST NOT** be summed with the four lifecycle-state quantities as if it were a fifth disjoint
   bucket; it partitions the existing `released` bucket.
4. Included in the idempotency canonical-equality comparison for `Release` and `ReleaseLease`: a replay supplying a
   different `release_outcome` than the originally
   accepted call **MUST** be rejected with `error_code = idempotency_conflict`.

Expiry-driven transitions (`expired`) carry no `release_outcome` because the caller never invoked `Release`
or `ReleaseLease`; such records remain distinguishable from caller-initiated releases by their terminal
lifecycle state, and any `release_outcome` observed on an `expired` record is an implementation defect
(shadow comparison **MUST** flag it). The `reconciled` audit flag of
`cpt-cf-licensing-service-fr-reconciliation` is orthogonal to `release_outcome`: a record may carry any
combination (e.g., `released = failed` with `reconciled = true`).

- **Rationale**: The lifecycle state `released` alone cannot distinguish "operation completed normally and returned
  unused capacity" from "operation failed mid-flight" or "operation was cancelled before execution". This information
  has direct operational value (reconciliation classification, consumer health monitoring, incident analysis) but does
  not affect admission, accounting, or recovery correctness; surfacing it as an explicit optional field preserves the
  clean decision contract without widening the lifecycle state machine or the shared recovery state machine.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-reconciliation-worker`,
  `cpt-cf-licensing-service-actor-platform-operator`

#### Admission Outcome and Decision Mapping

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-soft-hard-outcomes`

- **Priority**: p2
- **Strictness**: REQUIRED

Quota-governed operations **MUST** return one `admission_outcome` from the stable closed set: `admitted`,
`admitted_with_warning`, `admitted_with_cap`, `admitted_with_downgrade`, or `rejected`. The
`admission_outcome` **MUST** be returned in addition to, not in place of, the canonical `decision_outcome`
and `policy_effect` pair, and **MUST** map deterministically as follows:

- `admitted` **MUST** produce `decision_outcome = allow` with `policy_effect = none`.
- `admitted_with_warning` **MUST** produce `decision_outcome = allow` with `policy_effect = warned`.
- `admitted_with_cap` **MUST** produce `decision_outcome = allow` with `policy_effect = capped`.
- `admitted_with_downgrade` **MUST** produce `decision_outcome = allow` with `policy_effect = downgraded`.
- `rejected` **MUST** produce `decision_outcome = deny` with `policy_effect = none`.

The system **MUST NOT** return an `admission_outcome` whose mapping contradicts the returned
`decision_outcome` or `policy_effect`.

The `admission_outcome` closed set **MUST NOT** include a `deferred` variant. `policy_effect = deferred` and
primary `reason_code = quota_deferred` apply only to entitlement-layer responses and are specified in
`cpt-cf-licensing-service-fr-deferred-admission`; quota-layer delayed retry is expressed exclusively through
`admission_outcome = rejected` with an accompanying retry-after duration per
`cpt-cf-licensing-service-fr-retry-after-semantics`. Retry-after presence, the disjoint temporal /
non-temporal rejection classification (`R_temporal` / `R_non_temporal`), the `retry_after_seconds` type
contract, and the `max_retry_after_seconds` ceiling are specified in
`cpt-cf-licensing-service-fr-retry-after-semantics` and are not re-stated here.

##### Required contents of `contributing_reason_codes` per `admission_outcome`

The required contents of `contributing_reason_codes` (see `cpt-cf-licensing-service-fr-decision-contract`)
are:

- `admitted`: empty list.
- `admitted_with_warning`: contains `quota_warned` exactly once, regardless of how many applicable policies
  triggered the warn threshold; the set of contributing policies is recorded in structured explanation
  rather than in `contributing_reason_codes` itself.
- `admitted_with_cap`: contains `quota_capped` exactly once.
- `admitted_with_downgrade`: contains `quota_downgraded` exactly once; when the inner target evaluation
  itself admitted with cap or warn, the inner `reason_code` (`quota_capped` or `quota_warned`) is added as
  a second entry. No further entries are permitted.
- `rejected`: empty list when the rejection is direct (no downgrade attempted). When the admission engine
  attempted a downgrade path (per `cpt-cf-licensing-service-fr-soft-threshold-mechanics`) and the downgrade
  target itself rejected, the contributing list **MUST** contain exactly one entry,
  `quota_downgrade_attempted`, so that audit, shadow comparison, and operator tooling can distinguish
  "direct rejection" from "rejection after downgrade". No other entry is permitted on rejection.
  `quota_downgrade_attempted` is orthogonal to the temporal classification of the primary `reason_code`: it
  **MAY** appear alongside a primary `reason_code` from either `R_temporal` or `R_non_temporal` per
  `cpt-cf-licensing-service-fr-retry-after-semantics` (e.g. a downgrade target that rejects because the
  feature is not entitled yields `primary = feature_not_entitled`, `contributing =
  [quota_downgrade_attempted]`, and no `retry_after_seconds`). The presence of this marker does **not**
  imply retryability; retryability is determined solely by the primary `reason_code` set membership.

For `policy_effect = deferred` with `admission_outcome = allow`, `contributing_reason_codes` instead carries
exactly one `entitlement_pending_*` entry per `cpt-cf-licensing-service-fr-deferred-admission`. These two
specifications do not overlap: a deferred response has `admission_outcome = allow` and
`contributing_reason_codes = [entitlement_pending_*]`; all other `admission_outcome` values follow the
admission-outcome rules in this requirement.

`contributing_reason_codes` is lexicographically sorted ascending, deduplicated, and does not contain the
primary `reason_code`, per `cpt-cf-licensing-service-fr-decision-contract`.

- **Rationale**: Consumers need one canonical decision contract for every operation and a quota-specific
  admission label that does not redefine the canonical outcome semantics. The admission-outcome-to-decision
  mapping, the contributing-reason-codes construction rules, the retry-after mechanics, and the
  deferred-admission rules are distinct concerns and are specified in separate FRs so each can evolve
  independently; this FR owns the mapping and the contributing-reason-codes construction.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-service-integrator`

#### Retry-After Semantics

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-retry-after-semantics`

- **Priority**: p2
- **Strictness**: RELAXED

When `admission_outcome = rejected` (per `cpt-cf-licensing-service-fr-soft-hard-outcomes`), the presence of
`retry_after_seconds` on the decision response is governed by a closed mapping from the primary
`reason_code` of the rejection. The set `R_temporal` comprises exactly
`{ quota_rejected, limit_exceeded, rate_window_exhausted, budget_window_exhausted, concurrency_saturated }`;
the set `R_non_temporal` comprises exactly
`{ feature_not_entitled, offering_item_missing, scope_disabled, policy_denied, authority_data_stale }`.
Every rejection `reason_code` used in this PRD **MUST** belong to exactly one of these two disjoint sets;
extending either set requires a PRD update. `invalid_request`, `invalid_correlation_id`,
`invalid_subject_kind`, `invalid_subject_opaque_id`, and `conflicting_authority_configuration` are service
`error_code` values (see `cpt-cf-licensing-service-fr-reason-error-semantics`) returned through the
service-error contract and are **NOT** members of either rejection-`reason_code` set.

When `admission_outcome = rejected` with a `reason_code ∈ R_temporal` the response **MUST** include an
advisory `retry_after_seconds` duration; when the `reason_code ∈ R_non_temporal` the field **MUST NOT** be
included.

`retry_after_seconds` is an integer number of seconds (`uint32`) computed on the responding node against
`now_at_module` under the boundary-consistency rules in §4.4 Clock authority (persisted timestamps plus
bounded-skew local wall-clock); a value of `0` means "retry permitted immediately" (`0` is a valid value,
distinct from field absence). The module-wide ceiling `max_retry_after_seconds` MAY be lowered by a
per-quota-policy override; per-policy overrides **MUST** be integer seconds in the range
`[1, max_retry_after_seconds]` (the per-policy ceiling cannot be zero, but the per-response value still MAY
be `0` when the derivation formula yields zero at a window boundary). `max_retry_after_seconds` is a
module-wide startup-configurable constant: p1 default `3600`, allowed range `[1, 86_400]`; every node in one
deployment **MUST** use the same configured value. Derivation is quota-class-specific: fixed-window rate and budget
rejections derive `retry_after_seconds = ceil(window_end - now_at_module)` for the rejecting window; lease-based
concurrency rejection derives `retry_after_seconds = ceil(earliest_active_lease_expiry - now_at_module)` and
MAY therefore be `0` when no active lease still blocks re-entry.

`retry_after_seconds` is a **floor, not a deadline**: the caller **MUST NOT** retry earlier than the
advertised value and **MAY** retry later. Caller-side jitter discipline and the cross-policy
`max`-reduction used when multiple binding policies contribute temporal rejection causes are specified in
`cpt-cf-licensing-service-fr-soft-threshold-mechanics` (Hierarchical scope composition, Step 3c). Where the
service-error taxonomy of `cpt-cf-licensing-service-fr-reason-error-semantics` allows a service-error
`error_code` to carry `retry_after_seconds` (classes `capacity`, `dependency`, `internal`), the type,
clock source, zero semantics, and floor-not-deadline rule above apply identically; classes `client_error`
and `state_conflict` **MUST NOT** carry the field.

- **Rationale**: Retry-after presence, temporal / non-temporal classification, and the ceiling + override
  mechanism are a self-contained concern that governs how rejected decisions and retryable service errors
  are surfaced to consumer retry logic. Isolating the rules here lets admission-outcome mapping, deferred
  semantics, and soft-threshold composition each refer to a single canonical retry-after specification.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-service-integrator`

#### Deferred Entitlement Admission

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-deferred-admission`

- **Priority**: p2
- **Strictness**: RELAXED

`policy_effect = deferred` and primary `reason_code = quota_deferred` apply **only** to entitlement-layer
responses where the resolved entitlement grant is in a pending lifecycle state and **MUST NOT** be returned
by quota-layer admission responses; quota-layer delayed retry is expressed exclusively through
`admission_outcome = rejected` with an accompanying retry-after duration per
`cpt-cf-licensing-service-fr-retry-after-semantics`. The legacy-named primary `reason_code = quota_deferred`
is retained for stability of the closed set in `cpt-cf-licensing-service-fr-reason-error-semantics`; its
`quota_` prefix is legacy and does not reclassify `deferred` as a quota-layer effect.

##### Closed set of entitlement grant states that produce `deferred`

The closed set of entitlement grant states that **MUST** produce `policy_effect = deferred` is
`{ pending_activation, pending_provisioning, pending_approval, pending_payment }`. Any entitlement lifecycle
state outside this set **MUST NOT** produce `deferred`: `active` grants produce `decision_outcome = allow`
with `policy_effect = none` (unless modified by `capped`, `downgraded`, or `warned`); `expired`, `revoked`,
or otherwise `absent` grants produce `decision_outcome = deny`.

Licensing Service **MUST NOT** compute these four pending states itself; they are read as a grant-lifecycle
attribute reported by the resolving authority (native licensing store or legacy VendorA adapter) for the
scope under `cpt-cf-licensing-service-fr-authority-routing`. If the resolving authority does not report
pending lifecycle states at all, `policy_effect = deferred` is never produced for scopes routed to that
authority and active-versus-absent is the only binary the entitlement layer observes. If the authority
reports a lifecycle state that cannot be mapped to the four-element closed set above, the module **MUST**
return a service error with `error_code = authority_unavailable` for that request and **MUST NOT** silently
collapse the unknown state to any member of the closed set.

##### `contributing_reason_codes` on deferred

When `policy_effect = deferred` is returned, `contributing_reason_codes` (see
`cpt-cf-licensing-service-fr-decision-contract`) **MUST** contain exactly one entry from the closed set
`{ entitlement_pending_activation, entitlement_pending_provisioning, entitlement_pending_approval,
entitlement_pending_payment }` corresponding to the authority-reported grant state, alongside the primary
`reason_code = quota_deferred`, so consumers can distinguish the four pending causes without parsing
explanation strings. The same closed set pairs with primary `reason_code = feature_not_entitled` on
quota-layer rejection responses emitted by Rule 2 below — same exactly-one-entry rule, same
authority-reported-state derivation — so the pending-grant cause is preserved across the entitlement-layer
`deferred allow` and the quota-layer `rejected` rendering of the same underlying authority state. For every
other combination of `policy_effect` and primary `reason_code`, `contributing_reason_codes` **MUST NOT**
contain any `entitlement_pending_*` code.

##### `deferred_retry_after_seconds` mechanics

The response **MAY** include `deferred_retry_after_seconds` as an advisory ETA for when the grant is
expected to reach an active state; omitting this field means the ETA is unknown.
`deferred_retry_after_seconds` uses the same unit, type, and clock source as `retry_after_seconds` (integer
`uint32` seconds of server-side UTC wall-clock per
`cpt-cf-licensing-service-fr-retry-after-semantics`) but is bounded by a separate ceiling
`max_deferred_retry_after_seconds`. The absolute allowed range of this ceiling is `[1, 2_592_000]` seconds
(`30` days). The module-wide default value of `max_deferred_retry_after_seconds` is `604_800` seconds (`7` days).
On decision paths that resolve a quota policy (all quota-governed operations), per-quota-policy overrides of
`max_deferred_retry_after_seconds` **MUST** be honoured within the same absolute range, permitting shorter or
longer approval or provisioning horizons. On entitlement-only paths that do not resolve any quota policy (for example
`CheckFeatureEntitlement`, `GetEffectiveEntitlements`, `GetLicensingDecision` when the licensing decision is
produced without quota orchestration), only the module-wide default applies; per-policy overrides do not
participate. If the authority supplies an ETA exceeding the effective ceiling, the module **MUST** clamp
the returned value to that ceiling and emit a clamp-telemetry event; the original authority-supplied value
is preserved in audit records but **MUST NOT** be returned on the decision path.
`deferred_retry_after_seconds` is advisory only, **MUST NOT** be returned outside `policy_effect =
deferred`, and **MUST NOT** be used for server-side replay or admission enforcement.

Per the canonical mapping in `cpt-cf-licensing-service-fr-soft-hard-outcomes`, `policy_effect = deferred` is
returned only with `decision_outcome = allow`; callers interpret a deferred `allow` as "the business
entitlement exists but the supporting provisioning or approval is still being finalized". It is not a
silent denial.

##### Consumer action rules for `policy_effect = deferred`

To remove ambiguity about what a caller is permitted to do with a `deferred allow`, the following rules are
normative and **MUST** be honoured by every consumer module:

1. **Preview-only entitlement paths** (`CheckFeatureEntitlement`, `GetEffectiveEntitlements`,
   `GetEffectiveLimits`, `GetLicensingDecision`): the consumer **MUST NOT** execute the guarded business
   operation while `policy_effect = deferred`. The response is informational and the consumer SHOULD retry
   the entitlement check after `deferred_retry_after_seconds` elapses (or earlier at the consumer's
   discretion).
2. **Quota-governed paths** (`Check`, `CheckAndReserve`, `Commit`, `Release`, `Update`):
   `decision_outcome = allow` with `policy_effect = deferred` **MUST NOT** cause durable reservation state
   to be created, and **MUST NOT** cause a counter advance or absolute-level persistence through the
   `Update` path (`cpt-cf-licensing-service-fr-persisted-grant-update`). If the decision orchestrator would
   otherwise create a reservation or advance `committed_usage_in_window`, it **MUST** suppress that side
   effect and set `admission_outcome = rejected` with primary `reason_code = feature_not_entitled`
   (`R_non_temporal` per `cpt-cf-licensing-service-fr-retry-after-semantics`, so `retry_after_seconds`
   **MUST NOT** be emitted) on the quota response. The top-level `policy_effect` on this rejection is
   `none` per the `admission_outcome = rejected` mapping in
   `cpt-cf-licensing-service-fr-soft-hard-outcomes` and the `decision_outcome = deny` rule in
   `cpt-cf-licensing-service-fr-decision-outcome-semantics`. The pending-grant cause remains
   distinguishable from a terminal `feature_not_entitled` rejection via exactly one
   `contributing_reason_codes[entitlement_pending_*]` entry identifying the specific pending lifecycle
   state of the underlying grant; the entitlement-layer `deferred_retry_after_seconds` advisory (when
   present on the enclosing decision) is surfaced through the structured `decision_explanation` rather
   than as a `retry_after_seconds` field. Primary `reason_code = quota_deferred` **MUST NOT** appear on
   any quota-layer admission response, consistent with the opening paragraph of this FR. The consumer
   **MUST NOT** execute the guarded business operation in this state; retry SHOULD be delayed by
   `deferred_retry_after_seconds` when present.
3. **Commercial paths contingent on `pending_payment` / `pending_approval`**: a consumer **MUST NOT** start
   metered or chargeable work while the grant's lifecycle state is `pending_payment` or `pending_approval`.
   The entitlement layer continues to return `policy_effect = deferred`; the consumer module is responsible
   for holding the request until the grant reaches `active`. This rule realises the intent of "deferred
   allow is not silent denial" without giving consumers a license to execute billable work against a grant
   that has not yet activated.

Rule 2's quota-layer suppression is the single canonical bridge between entitlement-layer `deferred` and
quota-layer response shape; implementations **MUST NOT** return a `CheckAndReserve` or `Update` success
response (with `reservation_id`, or with a persisted counter advance) while the underlying entitlement
grant is in a pending state.

- **Rationale**: Deferred admission is a cross-cutting entitlement-layer concern with its own authority
  mapping, clamp rules, and consumer-action discipline; carving it out of the admission-mapping FR lets
  consumers reason about deferred separately from the rest of the admission contract and lets the
  entitlement-layer state machine evolve without destabilizing the quota-layer admission mapping.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-service-integrator`,
  `cpt-cf-licensing-service-actor-platform-operator`

#### Entitlement Grant Renewal Grace

- [ ] `p3` - **ID**: `cpt-cf-licensing-service-fr-entitlement-grant-renewal-grace`

- **Priority**: p3
- **Strictness**: RELAXED

The resolving authority (native licensing store or legacy adapter) **MAY** report an optional
`renewable_until: RFC3339_UTC_ms` attribute on an entitlement grant, indicating the wall-clock instant up to
which that authority will accept a renewal (re-activation) of the same commercial entitlement. `renewable_until`
is **informational only**. It **MUST NOT**:

1. Alter the closed set of entitlement-grant lifecycle states
   `{ active, pending_activation, pending_provisioning, pending_approval, pending_payment, expired, revoked,
   absent }` declared in `cpt-cf-licensing-service-contract-legacy-adapter`. A grant whose commercial window
   has ended and whose service has been deprovisioned reports `state = expired` regardless of whether
   `renewable_until` is present; a grant whose service continues while billing retries reports `state = active`
   regardless of whether `renewable_until` is present.
2. Alter admission, `decision_outcome`, `policy_effect`, primary `reason_code`, `contributing_reason_codes`, or
   `retry_after_seconds`. A decision produced against an `expired` grant remains `decision_outcome = deny` with
   primary `reason_code` drawn from the rejection set, irrespective of `renewable_until` state.
3. Participate in any capacity arithmetic, window counter, reservation bound, or shared recovery state machine
   transition.

When `renewable_until` is reported by the authority, the system **MUST** propagate it as a top-level optional
field `renewable_until` on the following response payloads, so consumers can drive renewal UX or background
renewal retries without parsing free-form explanation text:

- Entitlement lookup responses (`CheckFeatureEntitlement`, `GetEffectiveEntitlements`, `GetEffectiveLimits`,
  `GetLicensingDecision`) per `cpt-cf-licensing-service-interface-lookup-response` and
  `cpt-cf-licensing-service-interface-decision-response`.
- Quota Decision Response contract (`cpt-cf-licensing-service-interface-quota-decision-response`) when the
  quota path resolved an entitlement grant carrying the attribute.

Emission rules:

- The field **MUST** be omitted from the response when the resolving authority did not report it, or when the
  reported instant is not strictly later than `now_at_evaluation` (the grace window has already closed).
  Omitted and `null` are treated identically.
- When present, the value **MUST** be a wall-clock instant on the same UTC clock source and precision used for
  other server-assigned timestamps (RFC3339 UTC, millisecond precision; see §4.4 Clock authority).
- Shadow comparison **MUST** treat `renewable_until` as an **additive, non-comparative** field in p1: it is
  excluded from `cpt-cf-licensing-service-fr-shadow-observability` canonical-equality buckets so that two authorities
  that differ only on
  renewability semantics do not create mismatch noise during migration. Including `renewable_until` in shadow
  equality is an additive, non-breaking future extension.
- Shadow comparison **MUST** additionally flag the combination `state = active` with
  `renewable_until <= now_at_evaluation` as an authority-reporting defect (adapter post-condition violation),
  because an already-closed renewal window on an active grant is semantically undefined. The defect is reported
  through adapter health telemetry; the live decision is not altered.
- Audit records (`cpt-cf-licensing-service-fr-decision-contract`) **MUST** persist `renewable_until` when it
  was present on the decision, so operators can reconstruct why a consumer surfaced a renewal prompt.

Legacy authorities that cannot distinguish a renewal grace window (for example, legacy catalogues that track
only `active` / `expired` / `revoked`) **MUST** omit the field; Licensing Service then emits no
`renewable_until` for scopes routed to that authority, and consumers interpret the absence as "renewal grace
semantics unavailable from this authority" rather than as "renewal grace = 0".

- **Rationale**: Commercial licensing lifecycles in the industry (SaaS dunning, post-expiration renewal windows, offline
  payment grace) routinely need to distinguish "entitlement over, and renewable" from "entitlement over, and terminally
  closed". Widening the lifecycle state closed set to add a fifth state couples this orthogonal signal with every
  consumer of the state machine (recovery, shadow, reconciliation, audit). A purely informational attribute keeps
  admission, state machine, and shadow comparison unchanged while giving consumers enough information to drive renewal
  UX and back-office workflows.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-tenant-administrator`,
  `cpt-cf-licensing-service-actor-platform-operator`, `cpt-cf-licensing-service-actor-migration-engineer`

#### Preview versus Strict Reservation Semantics

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-preview-vs-reserve`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** expose quota preview (`Check`) and strict reservation (`CheckAndReserve`) as distinct operations
with distinct lifecycle effects. `Check` **MUST NOT** create durable reservation state, **MUST NOT** allocate a
reservation identifier, and **MUST NOT** mutate runtime counters. `CheckAndReserve` **MUST** create durable
reservation state and **MUST** transition runtime counters atomically with its decision.

Given identical inputs (same normalized licensing scope, same `client_operation_id`, same estimated resource
dimensions, same resolved authority mode) and the same observed runtime state at evaluation time (same
`outstanding_reserved_quantity_in_window`, same `committed_usage_in_window`, same `effective_limit`, same freshness
classification), `Check` and `CheckAndReserve` **MUST** produce the same `decision_outcome`, the same
`admission_outcome`, the same `policy_effect`, the same primary `reason_code`, the same `effective_limit[d]` per
resource dimension `d`, and the same `retry_after_seconds` (when the field is present). In particular, `Check`
**MUST** read the same combined `outstanding_reserved_quantity_in_window + committed_usage_in_window` that
`CheckAndReserve` reads when validating admission per
`cpt-cf-licensing-service-fr-reserved-consumed-independence`; an optimistic preview that considers only
`runtime_committed_quantity` is **NOT** permitted.

Two deltas between the two operations are explicitly permitted and are the only permitted differences:

1. **Side-effect delta**: `Check` does not create a reservation record, does not mutate counters, and does not
   return a `reservation_id`; `CheckAndReserve` does all three. `admitted_quantity` fields (from
   `admitted_with_cap`) are computed identically in both operations, but `Check` does not bind the admitted
   magnitude to any reservation.
2. **Degraded-state delta**: per `cpt-cf-licensing-service-fr-degraded-policy`, `Check` is in the degraded-eligible
   operation-class whitelist and **MAY** return `decision_state = degraded`; `CheckAndReserve` is not in that
   whitelist and in the same circumstances **MUST** return a service error instead of a degraded success result.
   In that scenario the two responses are formally different, but this asymmetry is the documented degraded-path
   behavior and does not constitute an equivalence violation.

Race conditions between a `Check` call and a subsequent `CheckAndReserve` call are out of scope of this
equivalence rule: the invariant constrains behavior "at identical state", not "across time". Consumers SHOULD
treat a `Check` result as advisory and anticipate that the follow-up `CheckAndReserve` may yield a different
outcome if state moved.

##### Idempotency semantics for `Check`

Consumers **MAY** supply `client_operation_id` and `idempotency_key` on `Check` to correlate the preview with
later `CheckAndReserve` / `Commit` / `Release` calls under the same `client_operation_id`, but **MUST NOT**
rely on `Check` to deduplicate invocations. Specifically:

1. `Check` **MUST NOT** create an idempotency record in the fast-path state store. Replays of a `Check` call
   with the same `(client_operation_id, operation_type = check, idempotency_key)` triple are not matched
   against any prior record; each invocation is evaluated fresh against current state.
2. `Check` **MUST NOT** return a cached response from an earlier `Check` invocation; every call re-evaluates
   admission from current counters, current reservations, and current entitlement/freshness state. Two calls
   seconds apart can therefore produce different outcomes and this is not a contract violation.
3. Supplying a `client_operation_id` on `Check` that was previously used by a `CheckAndReserve` is permitted;
   `operation_type = check` and `operation_type = check_and_reserve` own disjoint replay records per
   `cpt-cf-licensing-service-fr-idempotent-lifecycle`, so there is no conflict. A `Check` does not inherit
   or expose the earlier `CheckAndReserve`'s reservation identifier; `reservation_id` is never returned by
   `Check`.
4. `idempotency_key` supplied on `Check` **MUST NOT** be rejected as malformed under the usual validation
   rules, but the module is not required to persist or match it; it is visible only on structured logs and
   telemetry for correlation.

These rules make `Check` a pure read whose idempotency semantics differ from state-changing operations by
design; they realise the side-effect delta already documented above without requiring `Check` to reserve
durable storage for replay matching.

- **Rationale**: Preview operations are used for cost estimation and user-facing feedback; without a strict
  equivalence invariant at identical state, preview and strict admission silently drift and mislead consumers.
  The two documented deltas are required by existing invariants elsewhere in this PRD.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-service-integrator`

#### Explainable Decision Responses

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-explainable-decisions`

- **Priority**: p2
- **Strictness**: REQUIRED

The system **MUST** return explainable decision metadata that identifies the limiting scopes, relevant policy
identifiers, affected resource dimensions, decision timestamp, operation identity, and any retry-after or downgrade
instructions. Quota-layer explanation **MUST** be emitted through the structured `decision_explanation` payload
specified in `cpt-cf-licensing-service-fr-explanation`; implementations **MUST NOT** invent parallel free-form
explanation fields alongside it. `retry_after_seconds`, `deferred_retry_after_seconds`, `admitted_magnitude`,
and `target_policy_ids` remain top-level decision-response fields per
`cpt-cf-licensing-service-interface-quota-decision-response` and are not duplicated inside
`decision_explanation`.

- **Rationale**: Operators and consuming services must be able to explain quota outcomes to users, support teams, and
  auditors; a single structured payload is required for machine-comparable explanations.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`,
  `cpt-cf-licensing-service-actor-tenant-administrator`

#### Direct Update for Persisted Grants

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-persisted-grant-update`

- **Priority**: p2
- **Strictness**: RELAXED

The system **MUST** support a direct `Update` admission operation that advances per-window committed usage (for
counted dimensions) or persists a new per-subject level (for absolute dimensions) without requiring a prior
`CheckAndReserve`. `Update` is applicable **iff** every requested dimension's registered resource type satisfies
both `activity_type = persisted` and `quantity_model ∈ { counted, absolute }`. Requests that target
`activity_type = transient` dimensions, or `quantity_model = incremental` dimensions, **MUST** be rejected with
`error_code = invalid_request`; reservation-bounded pre-allocation remains the only correct admission sequence
for transient activity (because the guarded work is bounded in time and must not overshoot during execution)
and for incremental deltas (because a positive delta without an outstanding reservation would bypass the
admission bound established by `cpt-cf-licensing-service-fr-reserved-consumed-independence`).

`Update` **MUST**:

1. Require a stable `client_operation_id` and a transport-level `idempotency_key` per
   `cpt-cf-licensing-service-fr-idempotent-lifecycle`. The `operation_type` value is `update`, drawn from the
   Operation Type closed set. Transport-level replay records, canonical-equality rules, and conflict detection
   are identical to `Commit`; `operation_type = update` and `operation_type = commit` own disjoint replay
   records under the same `(client_operation_id, idempotency_key)` pair.
2. Evaluate admission against the same per-policy formula defined in
   `cpt-cf-licensing-service-fr-reserved-consumed-independence`. For `quantity_model = counted`, the admission
   check on every requested dimension `d` and every governing policy `p ∈ P_applicable` is
   `outstanding_reserved_in_window[p, d] + committed_usage_in_window[p, d] + requested_quantity[d] <=
   effective_limit[p, d]`. For `quantity_model = absolute`, the admission reduces to the per-subject version-
   guarded write and scope-level-sum check defined in `cpt-cf-licensing-service-fr-absolute-stale-write-
   protection`; `Update` is the public entry point for that path, and the semantics of monotonic version
   comparison, scope-level aggregation across `subject_scope` values, and zero-level pruning are unchanged.
3. On `admission_outcome ∈ { admitted, admitted_with_warning, admitted_with_cap, admitted_with_downgrade }`,
   atomically advance `committed_usage_in_window[p, d]` by the admitted magnitude (for counted) or persist the
   new level under a strictly greater version (for absolute). No reservation record is created, no
   `reservation_id` is returned, and no transition through the `reserved` lifecycle state occurs. Soft-threshold
   mechanics (`warn_threshold_percent`, `cap_enabled`, `downgrade_target_policy_id`) from
   `cpt-cf-licensing-service-fr-soft-threshold-mechanics` apply identically to counted `Update` admission; for
   absolute `Update` the same mechanics apply as already specified for absolute admission in that requirement.
4. On `admission_outcome = rejected`, produce no state change beyond the transport-level idempotency replay
   record. The rejection `reason_code`, `retry_after_seconds` (when `reason_code ∈ R_temporal`), and
   decision contract follow `cpt-cf-licensing-service-fr-soft-hard-outcomes` and
   `cpt-cf-licensing-service-fr-retry-after-semantics` identically to `CheckAndReserve` rejections.
5. Leave the enclosing `entitlement_grant` lifecycle state unchanged. `Update` mutates usage accounting, not
   commercial entitlement truth; an `Update` against a scope whose grant is not `active` is rejected through
   the entitlement layer under the same rules that reject `CheckAndReserve` in that state (including
   `policy_effect = deferred` suppression per `cpt-cf-licensing-service-fr-deferred-admission` Rule 2).
6. Be disjoint from any concurrent outstanding reservations or leases under the same scope and dimension. An
   active reservation or lease continues to occupy `outstanding_reserved_in_window` or `active_lease_count`
   and is neither collapsed, modified, nor settled by `Update`; settlement of a reservation remains the
   exclusive responsibility of `Commit` and `Release`, and of `ReleaseLease` for leases. Consumers **MUST
   NOT** use `Update` to short-circuit a reservation or lease they already hold.
7. Return the canonical Quota Decision Response defined in
   `cpt-cf-licensing-service-interface-quota-decision-response`. `reservation_id` and reservation lifecycle
   fields **MUST** be absent on `Update` responses because no reservation is created.

`Update` is idempotent under the same transport-level rules as `Commit`: a replay with the same
`(client_operation_id, operation_type = update, idempotency_key)` triple **MUST** return the previously
accepted result and **MUST NOT** advance counters or versions a second time. Cross-operation reuse between
`operation_type = update` and any other `operation_type` value under the same `client_operation_id` and
`idempotency_key` is **NOT** a replay and is **NOT** a conflict, per `cpt-cf-licensing-service-fr-idempotent-
lifecycle`.

- **Rationale**: Persisted activities (features enabled on a device, entitlements allocated to a subject, storage levels
  reported by a reporter) do not require reservation-based pre-allocation. Forcing every such consumer through
  `CheckAndReserve` + `Commit` is boilerplate that does not improve correctness, duplicates idempotency-record storage,
  and obscures the natural semantics of "advance the current count" or "set the current level". A direct `Update` path
  sharing admission math, idempotency rules, and decision contract with the reservation-based path preserves every
  admission and lifecycle invariant while simplifying the consumer contract for the persisted case.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-service-integrator`

### 5.8 Usage Accounting and Reconciliation

#### Durable Usage Accounting

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-durable-usage-accounting`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** maintain durable authoritative accounting separately from any fast-path enforcement state.

- p1 requires durable records for the simplified lifecycle in scope for p1 (`reserved`, `committed`, `expired`).
- p2 adds `released` records and operator-facing settlement visibility.
- p3 adds full audit overlays such as `reconciled` and deterministic post-failure reconstruction semantics.

- **Rationale**: Runtime counters alone are insufficient for auditability, reporting, and post-failure recovery.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`,
  `cpt-cf-licensing-service-actor-control-plane`

#### Idempotent Usage Lifecycle

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-idempotent-lifecycle`

- **Priority**: p1
- **Strictness**: REQUIRED

The system **MUST** provide idempotency for state-changing runtime operations, and the strength follows the p1/p2/p3
priority model.

- p1: the minimal runtime surface (`CheckAndReserve`, `Commit`) MUST use stable `client_operation_id` and
  `idempotency_key` values for key-based deduplication and best-effort replay. When a valid replay record
  exists, the previously accepted result MUST be returned instead of applying a second side effect.
- p2: stricter matching, propagation across accounting / audit surfaces, and later-tier runtime operations
  (`Release`, `Update`, lease operations) are added.
- p3: canonical request equality and deterministic replay semantics are required across structurally different
  but semantically equal payloads.
  Omitting required identifiers on a state-changing call in the currently active priority tier is `invalid_request`.
  Exact
  replay equality is defined over the client-supplied semantically relevant request fields of the addressed
  operation after the PRD's own default normalizations are applied. Server-assigned identifiers and timestamps are
  excluded from replay equality. Array order is significant unless a field's owning requirement declares a canonical
  sort rule, and numeric quantities are compared as integers in their declared canonical unit. A replay using the
  same `(client_operation_id, operation_type, idempotency_key)` triple with a canonically equal request **MUST**
  return the original outcome; the same triple with a canonically non-equal request **MUST** be rejected with
  `error_code = idempotency_conflict`. ADR-0004's stricter online TTL model applies from p2 onward.

- **Rationale**: Network retries, streamed workloads, and consumer recovery flows are normal platform behavior and must
  not corrupt quota state; stable operation identity is also required for cross-attempt correlation during recovery.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`

#### Online Idempotency and Archive Retention Separation

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-idempotency-vs-archive`

- **Priority**: p2
- **Strictness**: REQUIRED

The system **MUST** keep online replay protection logically separate from long-term archival.

- p1: replay protection may be implemented as a module-local best-effort TTL sufficient for operational safety.
- p2: explicit online-versus-archive separation and a module-wide TTL policy are required. The online replay
  horizon is `idempotency_ttl_seconds = max_reservation_ttl_seconds + settlement_grace_period_seconds`; a replay
  within that horizon returns the previously accepted result, while a replay after that horizon is rejected with
  `error_code = idempotency_record_expired`. Both terms are module-wide startup-configurable constants defined in
  the Glossary, and every node in one deployment **MUST** evaluate the same `idempotency_ttl_seconds` value.
- p3: replay retention may be tightened into deterministic audit / replay invariants.
  Historical archive retention is independent: `archive_retention_seconds` governs long-term storage only and
  **MUST NOT** be used as the online replay lookup horizon. Physical storage layout and retention enforcement
  mechanics are implementation details. ADR-0004 records the rationale for the module-wide TTL choice.

- **Rationale**: Online replay protection and historical audit retention have different availability, latency, and
  completeness requirements and cannot share a single storage path without ambiguous correctness semantics.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`

#### Reserved and Consumed Quantity Independence

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-reserved-consumed-independence`

- **Priority**: p2
- **Strictness**: RELAXED

The system **MUST** track `runtime_reserved_quantity` and `runtime_committed_quantity` as independent lifecycle
quantities and **MUST** validate admission decisions against the combined outstanding reservation plus committed
usage for the applicable scope and resource dimension, not against either quantity in isolation.
`runtime_reserved_quantity` and `runtime_committed_quantity` are **not** independent admission inputs: they
are usage-summary read-model aliases computed at the **record level** (not summed across policies), so
that each reservation record contributes its caller-supplied magnitude for dimension `d` exactly once
regardless of how many policies in `P_applicable` govern `d`. Concretely, `runtime_reserved_quantity[d]`
is the sum of `requested_quantity[d]` over all reservation records in state `reserved` whose
`accepted_at ∈ W_current` and for which at least one `p ∈ P_applicable` satisfies `governs(p, d) = true`;
`runtime_committed_quantity[d]` is the analogous sum of the record's committed magnitude for dimension
`d` over records in state `committed` whose `accepted_at ∈ W_current` under the same
applicable-policy filter. The filter decides which records are reportable at the scope's applicable
policy set; the magnitude is drawn from the record, so overlapping governance (multiple policies in
`P_applicable` governing the same `d`) does **not** double-count. Admission math never reads these
aliases and continues to use the per-policy per-window counters per the formulas below;
`cpt-cf-licensing-service-fr-usage-summary` reads the record-level aliases directly for operator
visibility.

For a committed record, its `committed_magnitude[d]` is the actual usage magnitude accepted by the `Commit`
operation for dimension `d` after all dimension-set, overshoot, and capacity checks have passed. When a commit is
accepted within an overshoot allowance, the persisted `committed_magnitude[d]` is that accepted actual value — it
is **NOT** clamped back down to `reserved_quantity[d]`.

The admission formula is stated per `quantity_model` and per quota class as follows:

- **`counted` and `incremental` (windowed)**: "outstanding reservation" and "committed usage" are scoped to the
  **current quota window** as defined in the Glossary entry *Current Quota Window* (half-open interval
  `[window_start, window_end)`). Reservations or committed-usage records from **closed prior windows** (Glossary) **MUST
  NOT** be
  counted against capacity in the current window, **regardless of whether the record is still in a non-terminal
  lifecycle state** (`reserved` with `accepted_at ∈ W_prior` and `expires_at ∈ W_current`). A reservation that
  straddles the boundary continues to count against the window identified by its `accepted_at` until it reaches
  a terminal state; capacity in `W_current` is unaffected by such a straddling record. Concurrency leases, which
  do not use windowed counters, are governed by the concurrency admission variant below. Admission admits iff,
  **for every applicable policy `p ∈ P_applicable` and every requested dimension `d` with `governs(p, d) = true`**,
  `requested_quantity[d] + outstanding_reserved_in_window[p, d] + committed_usage_in_window[p, d] <=
  effective_limit[p, d]` holds. `P_applicable` is the set resolved under
  `cpt-cf-licensing-service-fr-quota-policy-resolution`; the predicate `governs(p, d)` is defined in
  `cpt-cf-licensing-service-fr-soft-threshold-mechanics`. The degenerate case `|P_applicable| = 1` reduces to
  the single-policy form. Non-governing policies contribute no constraint on `d`. For each admitted `counted`
  event the per-window counter under the governing policy `p` **MUST** be incremented by `p.counted_unit`
  (see *Counted-unit policy field* below); for `incremental` the counter **MUST** be incremented by the
  caller-supplied `delta`. Per-policy windowed aggregation is keyed by `(normalized_licensing_scope,
  policy_id, window_start)`.
  For every formula in this PRD that refers to `remaining_capacity_under_p[d]` for a windowed policy,
  `remaining_capacity_under_p[d] = max(0, effective_limit[p, d] - outstanding_reserved_in_window[p, d] -
  committed_usage_in_window[p, d])`. The subtraction is evaluated in signed 128-bit intermediate precision and then
  saturated at `0`; if overshoot, reconciliation lag, or stale state would make the unsaturated difference negative,
  the canonical remaining capacity is still `0`.
- **`absolute`**: no window applies. The admission check is performed against the **current reported level**
  for the same protected object (see `cpt-cf-licensing-service-fr-absolute-stale-write-protection`).
  `requested_quantity` is not present on absolute-model writes; the write itself supplies the `level` and the
  admission condition is `scope_level_after <= effective_limit`, evaluated under the per-subject monotonic
  version check and the scope-level sum aggregation defined in that requirement.
- **Concurrency quotas (`cpt-cf-licensing-service-fr-concurrency-leases`)**: the protected capacity is active
  lease slots under the applicable scope, not reservations. `AcquireLease` carries
  `requested_slots: uint32` with default `1` and valid range `[1, 2^32 - 1]`. Concurrency admission is
  per-governing-policy exactly like the windowed variants: it admits iff, for every applicable policy `p` that
  governs the concurrency dimension, `requested_slots + active_lease_slots[p] <= effective_limit[p, d]` holds,
  where `active_lease_slots[p]` is the sum of `requested_slots` over leases currently in state `active` whose
  normalized licensing scope matches `p.scope_pattern` on that same dimension `d` (leases in state `released` or
  `expired` are excluded regardless of `reconciled` status). Concurrency admission does **NOT** use a time
  window and does **NOT** consume a reservation record; the lease is the admission artifact.

These three admission variants share the boundary-consistency rule in §4.4 Clock authority: all timestamp
comparisons (window attribution, lease expiry, absolute-write version ordering) read `now_at_evaluation`
as `now_at_module` on the evaluating node. Two concurrent nodes MAY disagree within
`±allowed_clock_skew_seconds` of an exact boundary; such bounded transient disagreement is tolerated and is
not an implementation divergence per §4.4.

##### Coverage invariant for requested dimensions

For every requested resource dimension `d` present on an admission-producing operation, `P_applicable`
**MUST** contain at least one quota policy `p` with `governs(p, d) = true`. Silent admission of an
ungoverned dimension as "implicitly unlimited" is forbidden. When the resolver produces a `P_applicable`
that covers only a strict subset of the requested dimensions, the module **MUST** return service error
`error_code = policy_config_conflict`; the response **MUST NOT** synthesize a `decision_outcome = allow` on
the basis of absent constraints. This invariant is symmetric with the existing rule that non-governing
policies contribute no constraint: absence of *any* governing policy is a configuration conflict, not a
constraint-free pass. Operators avoid the conflict by declaring a platform-level catch-all policy (either
explicitly finite or explicitly disabled via `policy_state = disabled`) that governs every declared
dimension of every registered resource type; the resolution invariant of
`cpt-cf-licensing-service-fr-quota-policy-resolution` then guarantees coverage for every admission.

##### Counted-unit policy field

Every quota policy with `quantity_model = counted` **MUST** declare `counted_unit: uint32` as a required
field. Valid range `[1, 2^32 - 1]`; values outside this range, and submissions that omit the field, are
rejected at policy admission with `error_code = policy_config_conflict`. There is **no** module-wide
default: absence of `counted_unit` on a `counted` policy is a configuration conflict, not a
silent-default condition. For `quantity_model = incremental` and `quantity_model = absolute`,
`counted_unit` **MUST NOT** be declared; such declarations are rejected with the same error code.
`counted_unit` affects admission arithmetic directly (per-admission counter advance) and therefore
participates in the determinism invariant of `cpt-cf-licensing-service-fr-quota-policy-resolution`: a policy
with a higher `counted_unit` consumes its window capacity faster than one with the default.

- **Rationale**: Validating only committed usage allows concurrent reservations to exceed limits; validating only
  reserved capacity ignores already-committed real consumption.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-reconciliation-worker`

#### Absolute-Value Stale-Write Protection

- [ ] `p3` - **ID**: `cpt-cf-licensing-service-fr-absolute-stale-write-protection`

- **Priority**: p3
- **Strictness**: RELAXED

For resource dimensions using `quantity_model = absolute`, the system **MUST** reject stale writes through an
optimistic-concurrency mechanism so that a stale reporter cannot overwrite a newer authoritative level. The
protected object is the full `normalized_licensing_scope` tuple (`(tenant_scope, subject_scope, product_family,
offering_item_or_feature, deployment_environment, usage_dimension)`) as defined in the Glossary; because
`subject_scope` is a component of that tuple, per-subject monotonicity is automatic and concurrent reporters
writing to different `subject_scope` values under an otherwise-equal tuple do not invalidate each other. A
write **MUST** be accepted only if its monotonic version is strictly greater than the currently persisted
version for that same protected object; any write with a version less than or equal to the persisted version
**MUST** be rejected. The scope-level quantity used for admission against `effective_limit` is the **sum** of
the latest accepted level taken across all values of `subject_scope` while the other five tuple components
(`tenant_scope`, `product_family`, `offering_item_or_feature`, `deployment_environment`, `usage_dimension`)
are held fixed; no other aggregation kind is supported in p1. When a dimension has no meaningful per-subject
breakdown, writers **MUST** use the canonical null-equivalent `subject_scope` value so that the aggregation
degenerates to a trivial single-subject case.

Per-subject levels are **not** automatically pruned in p1: to remove a subject's contribution from the scope
sum, the owning reporter **MUST** submit an explicit write with `level = 0` under a strictly greater version
than the last accepted write for that subject. Until such a zero-level write is accepted, the subject's
last-accepted level continues to contribute to the scope sum regardless of time elapsed since the last
write. Reconciliation-driven automatic pruning MAY be introduced in a future version; any such extension
requires a PRD update and an ADR.

Version field encoding is implementation detail, but the version check and admission evaluation **MUST** execute
atomically so that partial acceptance is never observable.

- **Rationale**: Absolute quantities report a level rather than a delta; without stale-write protection, out-of-order
  reporters can misstate quota and corrupt usage accounting.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`

#### Usage Summary and Remaining Quota Views

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-usage-summary`

- **Priority**: p2
- **Strictness**: REQUIRED

The system **MUST** provide aggregated usage summaries and remaining-quota views, and their fidelity follows the
p1/p2/p3 priority model.

- p1: basic remaining-capacity visibility is optional and not a priority exit criterion.
- p2: usage summaries distinguish the lifecycle states in scope for delivered runtime operations and expose
  enough data for operator interpretation.
- p3: summaries additionally expose reconciliation overlays and deterministic breakdown semantics without
  double-counting.
  At minimum, p2 summaries **MUST** keep the delivered lifecycle buckets distinct rather than collapsing them into
  one total: `reserved`, `committed`, `expired`, and, once release semantics are in scope, `released`; remaining-
  quota views **MUST** be derived from the same admission arithmetic that governs enforcement. At p3,
  `reconciled` is exposed only as an overlay or sub-breakdown on terminal records and **MUST NOT** be counted as a
  fifth disjoint lifecycle bucket. ADR-0003 records the rationale for the overlay model.

- **Rationale**: Governance, planning, and user-facing feedback require queryable aggregate state rather than raw
  lifecycle events only.
- **Actors**: `cpt-cf-licensing-service-actor-control-plane`, `cpt-cf-licensing-service-actor-tenant-administrator`,
  `cpt-cf-licensing-service-actor-consumer-module`

#### Reconciliation and Orphan Recovery

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-reconciliation`

- **Priority**: p2
- **Strictness**: REQUIRED

The system **MUST** provide reconciliation and orphan recovery once the module reaches operational maturity.
p1 does not guarantee reconciliation beyond reservation expiry. p2 introduces reconciliation workflows for
leaked reservations, stale aggregates, and settlement gaps. p3 adds durable reconciled-delta history and full
audit semantics, including the `reconciled` overlay described by ADR-0003.

- **Rationale**: Distributed failures and long-running operations create unavoidable divergence that must be corrected
  by the module rather than by manual operator action; a single-actor invariant on the audit flag ensures its
  semantic ("reconciled" vs "reconciliation ran") is unambiguous across implementations.
- **Actors**: `cpt-cf-licensing-service-actor-reconciliation-worker`, `cpt-cf-licensing-service-actor-platform-operator`

### 5.9 Reliability and Failure Handling

#### Concurrency Lease Safety

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-concurrency-leases`

- **Priority**: p2
- **Strictness**: REQUIRED

The system **MUST** support lease-based concurrency quotas in p2 and later for long-running operations whose hold
duration may exceed the reservation hard cap. p1 may defer this capability entirely. Once leases are in scope,
the module exposes `AcquireLease`, `RenewLease`, and `ReleaseLease`. Every concurrency-governing quota policy
**MUST** declare `default_lease_ttl_seconds` and `max_cumulative_lease_lifetime_seconds` as required integer-second
fields. `default_lease_ttl_seconds` has valid range `[1, max_cumulative_lease_lifetime_seconds]`; omission or an
out-of-range value is rejected at policy admission with `error_code = policy_config_conflict`. The per-policy field
`max_cumulative_lease_lifetime_seconds` is the canonical cumulative-lifetime ceiling (not a module-wide constant),
has valid range `[default_lease_ttl_seconds, 604_800]`, and omission or an out-of-range value is rejected with the
same error code. `AcquireLease` accepts optional `requested_ttl_seconds`; when present it is silently clamped to the
effective `default_lease_ttl_seconds` exactly analogously to `CheckAndReserve`. The lease lifecycle closed set is
`{ active, released, expired }`: `AcquireLease` creates `active`, `RenewLease` may extend only an `active`
lease, `ReleaseLease` transitions `active → released`, and expiry transitions `active → expired`. A `RenewLease`
request computes `candidate_expires_at` from the persisted current lease state and **MUST** satisfy
`candidate_expires_at - first_acquired_at <= max_cumulative_lease_lifetime_seconds` for every governing policy on
the lease's concurrency dimension; if any governing policy would be exceeded, the request **MUST** be rejected
with `error_code = lease_cumulative_lifetime_exceeded`. Lease operations use the same stable-operation identity and
replay rules as other state-changing runtime operations.

- **Rationale**: Concurrency governance fails if abandoned work can retain capacity indefinitely; long-hold
  workloads that exceed the reservation hard cap need a dedicated mechanism that still enforces a bounded
  cumulative lifetime and routes recovery through the same canonical state machine as reservations.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-reconciliation-worker`

#### Partial Failure Recovery

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-partial-failure-recovery`

- **Priority**: p2
- **Strictness**: REQUIRED

The system **MUST** preserve correct quota behavior when reservations succeed but downstream work never starts, when a
worker dies during execution, when usage delivery is delayed or replayed, and when a client retries after an unknown
outcome.

- **Rationale**: Quota correctness depends on reliable handling of partial failures, not only on successful request
  paths.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-reconciliation-worker`

#### Shared Recovery State Machine

- [ ] `p3` - **ID**: `cpt-cf-licensing-service-fr-shared-recovery-state-machine`

- **Priority**: p3
- **Strictness**: RELAXED

Reservation lifecycle transitions, lease lifecycle transitions
(`cpt-cf-licensing-service-fr-concurrency-leases`), and the post-terminal setting of the `reconciled` audit
flag **MUST** be enforced through one shared recovery state machine so that every consumer observes the same
valid transitions, invalid-transition rejections, and replay behavior. Consumer modules **MUST NOT** implement
ad-hoc reservation or lease recovery that bypasses these canonical transition rules. A consumer **MAY**
substitute an alternative implementation only if it preserves, at minimum, the conformance obligations below,
and the substitution is recorded in an ADR referenced from the consumer module's documentation and approved by
architecture review.

##### Conformance obligations

Any shared implementation or substitution **MUST** preserve at minimum:

1. Every valid reservation transition (`reserved → committed | released | expired`) under single-node and
   concurrent-retry scenarios, including canonical-equality replays.
2. Every invalid reservation transition (double-commit, double-release, settlement against terminal,
   commit-after-expiry, release-after-expiry) producing the additive error codes declared in
   `cpt-cf-licensing-service-fr-reservation-lifecycle`.
3. Every valid lease transition (`active → released | expired`; `active → active` on `RenewLease` within
   `max_cumulative_lease_lifetime_seconds`) and every invalid lease transition
   (`lease_expired`, `lease_already_released`, `lease_cumulative_lifetime_exceeded`).
4. Post-terminal setting of `reconciled` with the invariant that lifecycle state and quota capacity are
   unchanged.
5. Canonical equality of idempotent replays per `cpt-cf-licensing-service-fr-idempotent-lifecycle`.

A substitution passes only when it preserves the same externally observable outcomes for every conformance case.
Evidence of conformance **MUST** be revalidated on every major-version change of the shared recovery contract.
The code location, SDK packaging, and automated harness layout are implementation details.

- **Rationale**: Distributed recovery is only safe when every consumer follows the same transition rules; private
  recovery logic is the most common source of quota drift and double-settlement.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-service-integrator`,
  `cpt-cf-licensing-service-actor-reconciliation-worker`

#### Safe Degradation Policy

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-fr-safe-degradation`

- **Priority**: p2
- **Strictness**: RELAXED

The system **MUST** expose deterministic degraded-mode behavior that is policy-directed by operation and quota class so
consumer modules receive an explicit deny, allow, or emergency fallback outcome when quota dependencies are unavailable.
The system **MUST NOT** silently switch between fail-open and fail-closed behavior for the same policy context.
This FR applies only to operation classes that
`cpt-cf-licensing-service-fr-degraded-policy` permits to return degraded decisions. State-changing operations remain
governed by that FR's stricter rule and **MUST** return service errors instead of degraded success results.

##### Degraded-mode outcome alphabet

Every quota policy **MUST** declare a per-operation-class `degradation_policy` field with one value from the
closed set `{ fail_open, fail_closed, emergency_fallback }` whose mapping onto the canonical decision tuple
is exactly:

- `fail_closed`: when quota dependencies are unavailable, emit `decision_outcome = deny`,
  `policy_effect = none`, primary `reason_code = authority_data_stale`, `decision_state = degraded`. No
  admission side effects are produced. This is the default when `degradation_policy` is unset.
- `fail_open`: when quota dependencies are unavailable, emit `decision_outcome = allow`,
  `policy_effect = warned`, `admission_outcome = admitted_with_warning`, primary
  `reason_code = quota_warned`, `decision_state = degraded`. On the read / preview operation classes to which
  this FR applies, no durable reservation, counter, absolute-write, or lease side effect is produced; the
  response is advisory only. A future PRD version that makes a state-changing operation degraded-eligible
  **MUST** define that operation's degraded side effects locally rather than inheriting this rule.
- `emergency_fallback`: resolves the request against the active `emergency_override` record defined below.
  `emergency_override` records are not members of `P_applicable` and do not participate in normal admission
  arithmetic; they are consulted only after quota dependency failure on an operation class configured for
  `emergency_fallback`. When no `emergency_override` is active for the
  scope and resource dimension, `emergency_fallback` **MUST** degenerate to `fail_closed` (there is no
  permitted silent fall-open).

A given policy context **MUST** yield the same member of this triple on every invocation for which
dependencies are unavailable; switching between members without a configuration change is the
"silent switching" forbidden above. Additive extensions of this alphabet require a PRD update and an ADR.

##### Emergency override record and resolution

An `emergency_override` record has the closed schema: `override_id: String`, `scope_pattern` with the same grammar as
quota policies, `usage_dimension: String`, `expires_at: RFC3339_UTC_ms`, and optional `revoked_at: RFC3339_UTC_ms`.
Its decision semantics are fixed: when selected it yields `decision_outcome = allow`, `policy_effect = warned`,
`admission_outcome = admitted_with_warning`, primary `reason_code = quota_warned`, and `decision_state = degraded`.
An `emergency_override` is **active** iff its `scope_pattern` matches the request, its `usage_dimension` equals the
requested dimension, `revoked_at` is absent, and `expires_at > now_at_evaluation` under §4.4 Clock authority.
If multiple active overrides match, the selected record is the one with the narrowest scope per the canonical scope-
narrowness order; ties are broken by lexicographically ascending `override_id`. When selected, the override produces
the degraded decision above and **MUST NOT** create durable quota side effects on the operation classes governed by
this FR.

- **Rationale**: Expensive and risky operations need predictable failure semantics instead of panic-driven local
  improvisation during incidents.
- **Actors**: `cpt-cf-licensing-service-actor-consumer-module`, `cpt-cf-licensing-service-actor-platform-operator`

#### Soft Threshold, Cap, and Downgrade Mechanics

- [ ] `p3` - **ID**: `cpt-cf-licensing-service-fr-soft-threshold-mechanics`

- **Priority**: p3
- **Strictness**: RELAXED

The system **MUST** determine which `admission_outcome` to return from the closed set defined in
`cpt-cf-licensing-service-fr-soft-hard-outcomes` using three independent, per-quota-policy configuration fields,
each of which is optional with a safe default that preserves strict admission when unset:

- `warn_threshold_percent` (`uint8`, range `[0, 100]`, default unset): when set on a policy `p`, the warning
  signal for that policy fires iff the post-admission usage-over-limit ratio under `p`, reduced across the
  **governed requested dimensions** as `max_{d : governs(p,d) and d in request}(post_usage_ratio_under_p[d])`,
  reaches or exceeds `warn_threshold_percent / 100`. Dimensions not governed by `p` or not present in the
  request are excluded from the reduction; if no governed requested dimension exists, `p` contributes no
  warning signal. The comparison is performed using integer-only arithmetic to avoid floating-point drift: the
  firing condition is
  `(committed_usage[p,d] + outstanding_reserved[p,d] + requested_quantity[d]) * 100 >= warn_threshold_percent
    * effective_limit[p,d]`
  evaluated for the max-reducing dimension `d`. Overflow guard: the left-hand side is evaluated in 128-bit
  intermediate precision and the result is rejected with `error_code = invalid_request` if the intermediate
  product exceeds `2^127 - 1` before comparison. Division by zero: when `effective_limit[p,d] = 0`, policy `p`
  is treated as binding (admission is impossible under `p` for any non-zero request on `d`) and therefore
  contributes via the binding-set rules below, not via a warn signal; a zero-limit policy **MUST NOT** fire
  the warn signal. When unset, policy `p` contributes no warning signal.
- `cap_enabled` (`bool`, default `false`): governs per-policy willingness to admit a reduced magnitude when a
  request does not entirely fit under that policy. When the cap path admits (see composition rules below), the
  composed admitted magnitude **MUST** be returned to the caller so the caller can reserve exactly the admitted
  amount; a cap admission **MUST NOT** be returned with `admitted_magnitude[d] = 0` for any requested dimension
  `d` (see strict-cap rule below).
- `downgrade_target_policy_id` (optional policy identifier, default unset): governs per-policy willingness to
  re-route a not-fitting admission to a target policy. Maximum downgrade depth is `1` in p1: target policies
  **MUST NOT** themselves traverse their own `downgrade_target_policy_id` when invoked as a downgrade target,
  even if one is declared. In addition, the configured `downgrade_target_policy_id` graph across the full
  policy set **MUST** be acyclic at policy-admission time: any chain `p₀ → p₁ → … → p_k` with `p_k = p_i`
  for some `i ≤ k` (self-loops, two-cycles, and longer cycles) is rejected with `error_code =
  policy_config_conflict`, independently of whether the p1 depth-`1` runtime bound would prevent the cycle
  from activating. Chains with `k > 1` that are acyclic are also rejected at admission because the p1 depth
  bound would truncate them; operators MUST restructure such configurations before submission. Dangling
  targets (`downgrade_target_policy_id` pointing to a non-existent `policy_id`) are likewise rejected with
  `policy_config_conflict`.

##### Governs predicate

Let `governs(p, d)` be the boolean predicate on a quota policy `p` and a resource dimension `d` defined as:
`governs(p, d) = true` iff `p` declares a finite `effective_limit[p, d]` entry for dimension `d` in its
policy record and `p.scope_pattern` matches the request per the matching rule in *Scope Pattern*
(Glossary). The canonical on-wire authoring form of `effective_limit` is a **sparse map**:
`effective_limit: Map<usage_dimension, uint64>`. A dimension is non-governing exactly when its key is absent from
that map. `null`, `+∞`, `u64::MAX`, `0`-as-sentinel, and every other synthetic "unlimited" marker are forbidden on
the wire and **MUST** be rejected at policy admission with `error_code = policy_config_conflict`. Control-plane UIs
MAY expose an internal "unlimited" convenience to operators, but they **MUST** normalize it to key omission before
submitting the canonical policy payload. This predicate is normative for every formula below that ranges over
dimensions `d` — the min, max, and AND reductions **MUST** skip non-governing policies rather than substitute
sentinel values. In the canonical `effective_limits: Map<usage_dimension, uint64>` payload of
`cpt-cf-licensing-service-fr-explanation`, a dimension for which `governs(p, d) = false` **MUST** therefore be
absent from the map.

##### Hierarchical scope composition

Let `P_applicable` denote the set of applicable quota policies for the request across all scope levels evaluated
under `cpt-cf-licensing-service-fr-hierarchical-scope-evaluation` (platform, region, tenant, project, and user),
plus any policies that match registered subject kinds and product/resource filters through `scope_pattern`. Let
`P_binding ⊆ P_applicable` be the subset of policies under which the request does not entirely fit (i.e.,
`requested_quantity[d] > remaining_capacity_under_p[d]` for some requested dimension `d`). The degenerate
single-policy case `|P_applicable| = 1` follows these rules trivially.

Evaluation order **MUST** proceed deterministically as follows:

1. Compute, for each `p ∈ P_applicable`, the per-policy admission formula in
   `cpt-cf-licensing-service-fr-reserved-consumed-independence`. Derive `P_binding` from that computation.
2. **Fits everywhere** (`P_binding = ∅`): compute the per-policy warning signal for every `p ∈ P_applicable`
   whose `warn_threshold_percent` is set. If any policy contributes a warning signal, return
   `admitted_with_warning` with primary `reason_code = quota_warned`; the set of contributing policies is
   listed in the structured explanation (including their `policy_id` and scope level), and
   `contributing_reason_codes` is populated per `cpt-cf-licensing-service-fr-soft-hard-outcomes`. Otherwise
   return `admitted` with primary `reason_code = none`.
3. **Does not fit everywhere** (`P_binding ≠ ∅`):
   a. **Cap path**: if every `p ∈ P_binding` has `cap_enabled = true`, compute the composed admitted magnitude
   per requested dimension as
   `admitted_magnitude[d] = min_{p ∈ P_applicable : p governs dimension d}(remaining_capacity_under_p[d])`.
   The min is restricted to policies that actually govern dimension `d`; policies that do not govern `d`
   are excluded from that dimension's min because they never constrain `d`. By the coverage invariant
   of `cpt-cf-licensing-service-fr-reserved-consumed-independence`, every requested `d` has at least
   one `p ∈ P_applicable` with `governs(p, d) = true`, so the min is always well-defined: a request
   whose `P_applicable` covers only a strict subset of the requested dimensions is rejected with
   `error_code = policy_config_conflict` before this step is reached, and the "no governing policy for
   `d`" branch is therefore unreachable. If `admitted_magnitude[d] >= 1` for **every** requested
   dimension (strict-cap rule), return `admitted_with_cap` with that magnitude and primary
   `reason_code = quota_capped`. If any requested dimension would yield `admitted_magnitude[d] = 0`, cap
   is not available; fall through. If any `p ∈ P_binding` has `cap_enabled = false`, cap is not available;
   fall through.
   b. **Downgrade path**: if every `p ∈ P_binding` has `downgrade_target_policy_id` set, compose a recursive
   applicable set `P_applicable_inner = (P_applicable \ P_binding) ∪ P_targets`, where
   `P_targets = { resolve_policy(p.downgrade_target_policy_id) | p ∈ P_binding }`. This merge preserves
   every non-binding scope's hard constraints so that a downgrade cannot silently bypass a higher-scope
   ceiling that was already satisfied. Re-evaluate the request against `P_applicable_inner` at
   `depth = 1` using this same composition rule (without further downgrades). If the recursive evaluation
   admits (including with inner warn or cap), return `admitted_with_downgrade` with primary
   `reason_code = quota_downgraded`, `target_policy_ids` populated from `P_targets` (see ordering rule
   below), and `admitted_magnitude` carried through unchanged from the recursive evaluation (when the
   target evaluation itself admitted with cap, the target-level composed magnitude is the value carried).
   Any inner `quota_warned` or `quota_capped` signal produced at the target level is absorbed by the
   outer `admitted_with_downgrade` per the severity order below and contributes to
   `contributing_reason_codes` per `cpt-cf-licensing-service-fr-soft-hard-outcomes`. If any
   `p ∈ P_binding` has `downgrade_target_policy_id` unset, downgrade is not available; fall through.
   c. Otherwise, return `admission_outcome = rejected`. The primary `reason_code` on rejection is selected
   from the **most restrictive binding policy** under the following total order (evaluated per `p ∈
      P_binding`, where `P_binding` is computed in Step 1):

   i. **Normalized shortfall**: for each binding policy `p`, compute its per-dimension shortfall ratio
   `shortfall[p, d] = max(0, requested_quantity[d] - remaining_capacity_under_p[d]) * 10_000 /
   max(1, requested_quantity[d])` (non-negative 128-bit integer math; the division rounds toward zero,
   equivalently `floor` because all operands are non-negative; clamping the numerator at `0` keeps the ratio in the
   range `[0, 10_000]` basis points regardless of whether dimension `d` is itself binding under `p`)
   for every `d ∈ request` with `governs(p, d)`. Reduce across `d` as
   `policy_shortfall[p] = max_d shortfall[p, d]`. Because `p ∈ P_binding`, at least one governed
   dimension satisfies `requested_quantity[d] > remaining_capacity_under_p[d]`, so
   `policy_shortfall[p] > 0` is guaranteed. Larger `policy_shortfall` is more restrictive.
   ii. **Scope narrowness**: if two policies tie on `policy_shortfall`, the narrower scope wins per the
   canonical scope-narrowness total order
   (`user > project > tenant > region > platform`).
   iii. **`policy_id` lex ascending**: if two policies tie on both `policy_shortfall` and scope narrowness,
   the lexicographically smaller `policy_id` wins. This total order is deterministic by construction
   (no ties remain).

   The winning policy's `rejection_reason_code` is the primary `reason_code` on the response. When
   `reason_code ∈ R_temporal`, the advertised `retry_after_seconds` **MUST** be the
   `max` across the per-policy derivations of `retry_after_seconds` for every `p ∈ P_binding` with
   `p.rejection_reason_code ∈ R_temporal` (non-temporal binding policies contribute nothing to the max).
   Taking the max is the conservative choice: it guarantees the caller does not re-enter admission
   while any binding policy is still exhausted, avoiding retry storms that would otherwise hit the
   narrowest window early. The advertised value is a **floor, not a deadline**: the caller MUST NOT retry earlier
   than `retry_after_seconds`, and MAY retry later. Callers SHOULD add randomized jitter to the
   advertised value before scheduling the retry (the platform-recommended jitter is a uniform random
   addend in `[0, 0.2 * retry_after_seconds]` seconds) to de-synchronize the retry cohort that was
   rejected in the same tick; without jitter, a large fan-out of simultaneously-rejected callers would
   re-enter admission at the same instant and produce a second synchronized rejection burst. Jitter is
   a client-side obligation; the module does not randomize the advertised value because the floor
   semantics must remain deterministic for audit replay and shadow comparison.
   `rejection_reason_code` is an **optional per-quota-policy field** drawn from the closed set
   `R_temporal ∪ { policy_denied }` (i.e. the temporal rejection causes defined in
   `cpt-cf-licensing-service-fr-retry-after-semantics` plus the single non-temporal cause that a quota policy
   can legitimately produce). The remaining members of `R_non_temporal`
   (`feature_not_entitled`, `offering_item_missing`, `scope_disabled`, `authority_data_stale`) are
   entitlement-layer causes and **MUST NOT** be declared by a quota policy; doing so would make the
   quota layer impersonate an entitlement-layer denial and would contradict `limiting_layer = quota` in
   `decision_explanation`. When the policy does not declare `rejection_reason_code`, the module-wide
   default `quota_rejected` is applied. A policy **MUST NOT** declare `rejection_reason_code` as a value
   outside the permitted closed set; such declarations are rejected at policy admission with
   `error_code = policy_config_conflict`. Explanation metadata
   (`decision_explanation.contributing_policies`) lists **all** binding scopes regardless of which one
   won attribution.

The canonical scope-narrowness total order (narrowest → widest) used in step 3c and in any other rule in
this PRD that requires a tie-break across scope levels is:
`user > project > tenant > region > platform`. Scope levels outside this set **MUST NOT** be used in p1; a
future version MAY extend the order additively, but such extensions require explicit PRD update and an ADR.
When two binding policies share the same canonical scope level but differ only by domain-specific `subject_scope`,
`product_family`, `offering_item_or_feature`, or `usage_dimension` filters, those filters do not create additional
scope-narrowness levels; ties continue to fall through to `policy_id` lexicographic order after normalized
shortfall and canonical scope level have been compared.

When a single admission would satisfy more than one non-terminal modifier (for example, a would-be-capped
admission whose composed magnitude would also cross a `warn_threshold_percent` on another applicable policy),
the more-severe modifier wins per the severity order `capped > downgraded > warned` so that exactly one
`admission_outcome` is returned.

Soft-threshold mechanics (`warn_threshold_percent`, `cap_enabled`, `downgrade_target_policy_id`) apply to
`quantity_model ∈ {counted, incremental}` only. For `quantity_model = absolute`, the admission decision is
binary (accepted or rejected) per `cpt-cf-licensing-service-fr-absolute-stale-write-protection`, and `cap`
and `downgrade` are inapplicable (there is no reducible magnitude to admit and no meaningful re-route
target). `warn_threshold_percent` on an absolute policy is evaluated against the post-admission scope-level
ratio `scope_level_after / effective_limit` instead of the per-request `post_usage_ratio`; when it crosses,
`admission_outcome = admitted_with_warning` is emitted with the accepted level, and `admitted_magnitude`
and `target_policy_ids` **MUST NOT** be returned for absolute-model responses.

The composed admitted magnitude **MUST** be returned in the quota decision response as the field
`admitted_magnitude: Map<dimension, uint64>` (required when `admission_outcome = admitted_with_cap` for a
counted/incremental quota, carried through when `admission_outcome = admitted_with_downgrade` for such a
quota, and absent otherwise). The downgrade-target attribution **MUST** be returned as
`target_policy_ids: List<policy_id>` (non-empty when `admission_outcome = admitted_with_downgrade`, absent
otherwise) and **MUST** be sorted lexicographically ascending by `policy_id` string value and deduplicated,
so that two implementations produce byte-identical audit records and replay comparisons for the same
decision. Insertion order, scope-order, and any other non-canonical ordering are forbidden. Both fields are
part of the Quota Decision Response Contract in §7.1.

Threshold evaluation and any resulting reservation creation, counter mutation, or absolute-level persistence
**MUST** be atomic; partial state **MUST NOT** be observable.

- **Rationale**: Non-terminal admission modifiers are business-level, not implementation details; consumers must
  be able to predict exactly which modifier applies without reading the admission-engine code.
- **Actors**: `cpt-cf-licensing-service-actor-platform-operator`, `cpt-cf-licensing-service-actor-consumer-module`,
  `cpt-cf-licensing-service-actor-service-integrator`

#### Bounded Overshoot Declaration

- [ ] `p3` - **ID**: `cpt-cf-licensing-service-fr-bounded-overshoot`

- **Priority**: p3
- **Strictness**: RELAXED

The system **MUST** define and expose an explicit overshoot bound, or an explicit zero-overshoot requirement, for each
supported quota class and enforcement mode used by the platform. The overshoot bound is a per-quota-policy
allowance expressed as a percentage of the reserved quantity (field `allow_overshoot_bound_percent`, `uint8`,
range `[0, 100]`, default `0` which denotes "zero overshoot"). The per-dimension allowed magnitude at commit
time is derived from the reservation record's persisted `reserved_quantity[d]`, which is the admitted magnitude
actually bound at reserve time (for `admitted_with_cap`, the admitted capped magnitude; otherwise the accepted
requested quantity). For each governing policy `p` on dimension `d`, compute
`allowed_at_commit_under_p[d] = reserved_quantity[d] + ⌈reserved_quantity[d] × p.allow_overshoot_bound_percent / 100⌉`,
with integer-only arithmetic in 128-bit intermediate precision to avoid floating-point drift: the product
`reserved_quantity[d] × p.allow_overshoot_bound_percent` is evaluated first, then integer-divided by `100` with
remainder-based ceiling (equivalently `(product + 99) / 100` for non-negative integer inputs). The effective
`allowed_at_commit[d]` is the `min` of `allowed_at_commit_under_p[d]` across all governing policies for `d`
(the most restrictive overshoot allowance wins). A request in which any intermediate product exceeds `2^127 - 1`
before comparison is rejected with `error_code = invalid_request`. When every governing policy has
`allow_overshoot_bound_percent = 0`, `allowed_at_commit[d] = reserved_quantity[d]` exactly (no overshoot). A
commit whose actual usage exceeds this bound on any dimension **MUST** be rejected with
`error_code = commit_exceeds_reservation`; a commit within the bound but which would still violate any governing
policy after replacing the reservation hold with the committed magnitude **MUST** be rejected with
`error_code = overshoot_capacity_exhausted`. Commit-time overshoot evaluation, counter draw, persistence of
`committed_magnitude[d]`, and any resulting rejection **MUST** be atomic with the `committed` state transition.

- **Rationale**: Approximate enforcement is only governable when tolerated divergence is declared instead of being
  implicit.
- **Actors**: `cpt-cf-licensing-service-actor-platform-operator`, `cpt-cf-licensing-service-actor-control-plane`

### 5.11 Migration and Shadowing

#### Shadow Comparison Mode

- [ ] `p3` - **ID**: `cpt-cf-licensing-service-fr-shadow-mode`

- **Priority**: p3
- **Strictness**: REQUIRED

The system **MUST** support shadow mode in which non-authoritative native decisions are computed and compared against
authoritative legacy outcomes without affecting live decisions. In `shadow` mode, the authoritative decision **MUST** be
the only user-visible decision, and the shadow decision **MUST NOT** alter the returned allow or deny outcome,
effective-limit context, or service-error semantics for the live request.

- **Rationale**: Migration requires safe comparison before cutover.
- **Actors**: `cpt-cf-licensing-service-actor-migration-engineer`, `cpt-cf-licensing-service-actor-platform-operator`

#### Mismatch Visibility

- [ ] `p3` - **ID**: `cpt-cf-licensing-service-fr-shadow-observability`

- **Priority**: p3
- **Strictness**: REQUIRED

The system **MUST** record and expose the outcome of every authoritative-versus-shadow comparison (including
matches, so that `mismatch_rate` has a well-defined denominator) with enough context to debug model or mapping
differences. Each comparison record **MUST** include a `correlation_id`, authoritative and shadow reason codes,
authoritative and shadow decision states, and a `shadow_comparison_bucket` from the closed set
`{match, outcome_mismatch, effect_mismatch, reason_mismatch, limit_mismatch, shadow_error, shadow_timeout,
shadow_skipped}`. `match` records a successful equivalence of the two decisions; `outcome_mismatch` records a
difference in `decision_outcome`; `effect_mismatch` records a difference in `policy_effect` or `admission_outcome`
at equal `decision_outcome`; `reason_mismatch` records a difference in primary `reason_code` at equal outcome and
effect; `limit_mismatch` records a difference in `effective_limit` at equal outcome, effect, and reason, and
**MUST** be recorded only when both authoritative and shadow decisions produce effective limits for the same
canonical usage dimension but disagree on the canonical integer value; `shadow_error` records a non-timeout
failure of the shadow path; `shadow_timeout` records a shadow path that did not respond within its configured
deadline; `shadow_skipped` records the shadow path not being invoked (sampling, per-scope disablement, or no
shadow authority configured). Each record **MUST** carry exactly one bucket, selected by the severity order
`outcome_mismatch > effect_mismatch > reason_mismatch > limit_mismatch > shadow_error > shadow_timeout > shadow_skipped > match`.
Shadow comparison **MUST** be performed only
across the same normalized licensing scope, the same usage dimension, and canonicalized effective-limit units.
Absence of an authoritative decision is not a shadow comparison outcome; live-path authoritative failure is
reported as a service error through the normal decision path and is excluded from the shadow comparison
denominator.

**Shadow action policy**: shadow comparison is **observation-only**. The module emits structured
records per the schema above, exposes aggregate telemetry (`mismatch_rate` per
`shadow_comparison_bucket`, windowed per `normalized_licensing_scope`), and surfaces per-request buckets on
audit records keyed by `correlation_id`. The module **MUST NOT** take any automated action based on bucket
frequency, bucket severity, or `mismatch_rate` threshold: it **MUST NOT** switch `resolved_authority_mode`,
**MUST NOT** escalate the live decision's `decision_state` to `degraded`, **MUST NOT** reject the live
decision, and **MUST NOT** alter any subsequent admission. Operator-driven remediation (policy update,
authority-mode reconfiguration via the authority resolver, or migration rollback) is the sole response
path; operators consume the telemetry to make these decisions out-of-band. Automated response based on
mismatch telemetry (adaptive re-routing, kill-switches, circuit breakers) is out of scope for this PRD
unless introduced by a new functional requirement and an ADR.

- **Rationale**: Shadowing is only useful if divergences are measurable and actionable. Here, "actionable"
  means operator-actionable via telemetry; automated responses are deferred so that shadow comparison itself
  cannot become a source of unreviewed behavior change during migration.
- **Actors**: `cpt-cf-licensing-service-actor-migration-engineer`, `cpt-cf-licensing-service-actor-platform-operator`

### 6.1 Module-Specific NFRs

#### Warm-Path Criteria

Latency NFRs in this section apply only to **warm-path** requests. A request is **warm-path** iff all of the
following conditions hold for its evaluation:

1. The authority resolution cache contained an entry for the normalized licensing scope at request entry
   (no authority-rule table reload was triggered by this request).
2. Entitlement lookup was satisfied by the local projection (for legacy-authoritative scopes) or the native
   store's hot read path (for native-authoritative scopes) without triggering an upstream authority refresh
   or synchronous legacy VendorA adapter RPC.
3. Quota decisioning was satisfied by the fast-path state store for counter reads, reservation reads, and
   idempotency-record reads without falling back to the durable accounting store.
4. `freshness_state ∈ { fresh, stale_within_policy }` at response emission.
5. No outbound RPC to a legacy VendorA adapter, native-store refresh, or external cache tier was triggered
   by this request.

Requests that fail any of these conditions are **cold-path**; the latency thresholds below do not apply to
cold-path requests. Cold-path latency is bounded only by the availability threshold
(`cpt-cf-licensing-service-nfr-availability`) and by the staleness-visibility and degraded-policy
requirements. Implementations **MUST** emit a `path_class ∈ { warm, cold }` telemetry tag on every decision
so that p95 latency dashboards can compute the warm-path denominator deterministically.

#### Decision Latency

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-decision-latency`

The system **MUST** complete canonical entitlement lookup and effective-limit lookup operations within 50 ms at p95 for
warm-path requests backed by a fresh local projection or native store.

- **Threshold**: `<= 50 ms p95` for warm-path read operations using fresh local authority data.
- **Rationale**: Hot-path gating and quota orchestration require low-latency entitlement resolution.
- **Architecture Allocation**: Implementation allocation is captured in companion design artifacts.

#### Decision Availability

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-availability`

The system **MUST** provide deterministic outcomes for 99.9% of monthly licensing decision requests, including explicit
degraded decisions when policy permits non-authoritative evaluation and documented service errors when a valid decision
cannot be produced.

- **Threshold**: `99.9%` of monthly externally visible decision-producing requests at the licensing-service boundary
  return either authoritative decisions, policy-defined degraded decisions, or documented service errors with fixed
  `error_code` semantics. Internal retries, projection refresh operations, and shadow sub-requests are excluded from
  this denominator.
- **Rationale**: Platform modules require predictable behavior even during authority outages or projection lag.
- **Architecture Allocation**: Implementation allocation is captured in companion design artifacts.

#### Observability

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-observability`

The system **MUST** emit structured telemetry for authority routing, legacy projection freshness, native decisioning,
and all delivered runtime operations. When later-tier capabilities such as shadow comparison or reconciliation are in
scope, telemetry **MUST** extend to them as well.

- **Threshold**: All decision-producing operations emit `correlation_id`, authority mode, decision outcome, decision
  state, primary reason code, freshness metadata, and limiting-factor metadata. When shadow comparison is in scope,
  emitted comparison records additionally carry `shadow_comparison_bucket`.
- **Rationale**: Operators and migration engineers need consistent visibility into routing behavior and mismatch causes.
- **Architecture Allocation**: Implementation allocation is captured in companion design artifacts.

#### Auditability

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-auditability`

The system **MUST** retain enough decision and routing metadata to reconstruct why an entitlement or limit decision was
produced for at least 30 days in production environments.

- **Threshold**: Decision records retained for `>= 30 days` include at minimum `correlation_id`, licensing scope,
  resolved authority mode, decision outcome, decision state, primary reason code, freshness state, and source version or
  watermark when available.
- **Rationale**: Tenant support, compliance investigations, and migration validation need post-factum decision
  reconstruction.
- **Architecture Allocation**: Implementation allocation is captured in companion design artifacts.

#### Correlation and Traceability

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-correlation-traceability`

The system **MUST** propagate the same `correlation_id` across decision responses, service-error responses, audit
records, observability events, and any shadow comparison records generated by the same logical request when that
capability is in scope.

- **Threshold**: `100%` of decision-producing requests emit one stable `correlation_id` across response and emitted
  records.
- **Rationale**: Multi-system review and incident analysis require end-to-end traceability for a single licensing
  decision.
- **Architecture Allocation**: Implementation allocation is captured in companion design artifacts.

#### Single-Authority Safety

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-no-double-truth`

The system **MUST** enforce configuration and validation rules that prevent one licensing scope from having more than
one authoritative source at the same time.

- **Threshold**: All configuration changes that would create dual-authoritative scope ownership are rejected.
- **Rationale**: Conflicting business truth creates inconsistent entitlements and unsafe migration behavior.
- **Architecture Allocation**: Implementation allocation is captured in companion design artifacts.

#### Staleness Visibility

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-staleness-visibility`

The system **MUST** expose explicit stale-state signals and apply the configured degraded-mode policy rather than
silently serving stale data as fresh when projection or cache staleness exceeds configured policy thresholds.

- **Threshold**: Any response served after freshness threshold breach includes explicit stale-state metadata and
  degraded-mode semantics.
- **Rationale**: Consumers and operators must be able to distinguish fresh authoritative data from tolerated stale
  reads.
- **Architecture Allocation**: Implementation allocation is captured in companion design artifacts.

#### Quota Decision Latency

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-quota-decision-latency`

The system **MUST** complete quota preview and `CheckAndReserve` operations within 50 ms at p95 for warm-path
requests when all required policy, counter, and entitlement data is locally available.

- **Threshold**: `<= 50 ms p95` for warm-path quota decisions.
- **Rationale**: Quota gating must not introduce user-perceptible latency for interactive API requests.
- **Architecture Allocation**: Implementation allocation is captured in companion design artifacts.

#### Reservation and Settlement Durability

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-reservation-durability`

The system **MUST** guarantee that no acknowledged settlement record for the lifecycle states delivered in the active
priority tier is lost once acknowledged to the caller. In p1 this applies to committed settlement on the minimal runtime
surface; later tiers extend the guarantee to additional terminal outcomes such as `released`.

- **Threshold**: Zero acknowledged settlement records lost under single-node failure.
- **Rationale**: Billing, audit, and reconciliation correctness depend on durable usage accounting.
- **Architecture Allocation**: Implementation allocation is captured in companion design artifacts.

#### Reconciliation Lag

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-reconciliation-lag`

When reconciliation workflows are in scope, the system **MUST** reconcile orphaned reservations and mismatched usage
within a configurable maximum lag bound. This is not a p1 exit criterion.

**Normal operation** for the purpose of this threshold means all three of the following hold: (a) no
platform-declared incident is active against the licensing-service component; (b) the reconciliation-worker
backlog (count of terminal records whose `terminal_at` is older than
`max_reservation_ttl_seconds + settlement_grace_period_seconds` and whose `reconciled` audit flag is still
`false`, where `terminal_at` is the persisted record-creation timestamp of the terminal lifecycle transition —
namely `committed_at` for records terminal in state `committed`, `released_at` for `released`, and
`expired_at` for `expired` — written exactly once by the fast-path state store per §4.4 Clock authority rule
(1), and likewise `committed_at`/`released_at`/`expired_at` for lease records)
is below the configured alert threshold; (c) both the fast-path state store and the durable
accounting store report healthy on their platform health checks. The compound horizon
`max_reservation_ttl_seconds + settlement_grace_period_seconds` matches the lower bound defined in
`cpt-cf-licensing-service-fr-idempotency-vs-archive`, so a record still unreconciled beyond it has
demonstrably outlived every replay-safe settlement path. Outside normal operation
the `<= 5 min` bound is a Service Level Objective and **MUST NOT** be interpreted as a hard runtime
invariant; the reconciliation-lag metric **MUST** continue to be emitted so operators can quantify the
breach.

- **Threshold**: `<= 5 min` reconciliation lag under normal operation as defined above; alert on configurable threshold
  breach.
- **Rationale**: Stale reservation state blocks valid work and produces inaccurate remaining-quota views.
- **Architecture Allocation**: Implementation allocation is captured in companion design artifacts.

#### Quota Observability

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-quota-observability`

The system **MUST** emit structured telemetry for all delivered quota decisions, reservation lifecycle transitions,
settlement events, and policy changes. When reconciliation is in scope, the emitted telemetry **MUST** extend to
reconciliation actions as well.

- **Threshold**: All quota operations emit operation identity, subject, scope, affected resource dimensions, decision
  outcome, limiting scopes, and reservation lifecycle state.
- **Rationale**: Operators, tenant administrators, and support teams need consistent visibility into quota behavior
  across workloads.
- **Architecture Allocation**: Implementation allocation is captured in companion design artifacts.

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-contract-legacy-adapter`

## 8. Use Cases

Optional: Include when interaction flows add clarity beyond requirement statements. Use cases below are examples and
follow the same priority rules as their owning requirements.

#### Check Feature Entitlement for a Module Request

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-usecase-check-feature-entitlement`

- **Priority**: p1

**Actor**: `cpt-cf-licensing-service-actor-consumer-module`

**Preconditions**:

- Licensing scope is derivable from request context.
- Licensing authority data is available from either a fresh projection or native store.

**Main Flow**:

1. The consumer module calls the canonical licensing API with subject, feature, and scope context.
2. The licensing service resolves authority mode for the scope.
3. The licensing service loads canonical entitlement and effective-limit context.
4. The licensing service returns a decision with `decision_outcome`, `decision_state`, primary `reason_code`,
   freshness metadata, and `correlation_id`. Later tiers may additionally return `policy_effect` and structured
   explanation payloads.

**Postconditions**:

- The consumer module receives an explainable licensing decision.

**Alternative Flows**:

- **Authority data stale or unavailable**: The licensing service returns a policy-defined degraded or denied outcome
  with stale-state metadata.
- **Invalid request or unresolved scope**: The licensing service returns a service error with fixed `error_code`
  semantics instead of a business decision.

#### Quota-Governed API Request (Reserve, Execute, Settle)

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-usecase-quota-governed-request`

- **Priority**: p1

**Actor**: `cpt-cf-licensing-service-actor-consumer-module`

**Preconditions**:

- Quota policy exists for the relevant scope and resource dimensions.
- Entitlement and effective-limit data is available.

**Main Flow**:

1. The consumer module calls `CheckAndReserve` with a `client_operation_id`, an `idempotency_key`, execution
   context, and estimated resource dimensions.
2. The module internally resolves entitlement, effective limits, and applicable quota policies.
3. In p1 the module evaluates the minimal quota policy needed for the request. Later tiers may add
   multi-dimensional and hierarchical policy behavior.
4. When the response carries `decision_outcome = allow`, a reservation is created and returned with reservation
   identifier, state, and expiry. Later tiers may additionally classify the allow path with `admission_outcome`
   and related soft-outcome metadata.
5. The consumer module executes the operation.
6. The consumer module calls `Commit` with actual usage dimensions and the same `client_operation_id`.
7. The module settles usage and updates durable accounting. Later tiers may add explicit release and reconciliation
   behavior beyond reservation expiry.

**Postconditions**:

- The request outcome is explained by durable quota state and remaining quota is updated.

**Alternative Flows**:

- **Hard limit exceeded**: The module returns `decision_outcome = deny`; later tiers may additionally expose
  `admission_outcome`, limiting-scope metadata, and retry-after guidance.
- **Soft limit triggered**: Soft outcomes such as capping, downgrade, warning, or deferral are later-tier behavior and
  are not p1 exit criteria.
- **Consumer failure before commit**: In p1 the reservation expires; later tiers add reconciliation-driven recovery.

#### Long-Running Job Concurrency Control

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-usecase-long-running-job`

- **Priority**: p2

**Actor**: `cpt-cf-licensing-service-actor-consumer-module`

**Preconditions**:

- A consumer module is about to start a long-running job that consumes concurrency capacity.
- Concurrency quota policy exists for the relevant scope.

**Main Flow**:

1. The consumer module requests capacity for a long-running slot.
2. The module creates lease-backed state only in priority tiers where lease semantics are delivered.
3. The job starts and the consumer module maintains lifecycle progress.
4. When the job completes, the consumer module settles the slot according to the active lease contract.
5. If the consumer module fails, expiry and reconciliation recover orphaned capacity.

**Postconditions**:

- Active capacity reflects completed work and leaked slots are recoverable through reconciliation.

**Alternative Flows**:

- **No capacity remaining**: The module returns `decision_outcome = deny` with `admission_outcome = rejected` and the
  job is not started.
- **Worker failure**: Reservation expires and reconciliation restores capacity.

#### Policy Administration and Effective Quota Inspection

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-usecase-policy-admin`

- **Priority**: p2

**Actor**: `cpt-cf-licensing-service-actor-control-plane`

**Preconditions**:

**Main Flow**:

1. The control plane lists or updates quota policies for a selected scope.
2. The module validates and persists the policy change.
3. In later tiers, the control plane may request effective quota inspection for a subject and action context.

**Alternative Flows**:

#### Compare Shadow Decisions During Migration

- [ ] `p3` - **ID**: `cpt-cf-licensing-service-usecase-compare-shadow-decisions`

- **Priority**: p3

**Actor**: `cpt-cf-licensing-service-actor-migration-engineer`

**Preconditions**:

- The licensing scope is configured in `shadow` mode.
- Both authoritative legacy and shadow native decision paths are available.

**Main Flow**:

1. A licensing decision request is processed using the authoritative legacy path.
2. The licensing service computes the shadow native decision for the same input.

## 9. Acceptance Criteria

Business-level acceptance criteria for the PRD as a whole.

Acceptance is priority-scoped: a p1 delivery is judged primarily against functional requirements tagged
`Priority: p1` with `Strictness: REQUIRED`. Deferred bullets apply only when their owning FR priority tier is active.

#### p1 Minimum Acceptance

- [ ] A decision-producing p1 operation returns the minimal canonical decision contract: `decision_outcome`,
  `decision_state`, resolved authority mode, normalized licensing scope, primary `reason_code`, `freshness_state`, and
  `correlation_id`, with service errors kept separate from business decisions.
- [ ] `Check` does not create durable reservation state or mutate runtime counters; `CheckAndReserve` creates durable
  reservation state; `Commit` settles or rejects against the existing reservation contract.
- [ ] p1 state-changing operations require stable `client_operation_id` and `idempotency_key`; replay returns the
  previously accepted result and does not duplicate side effects.
- [ ] Invalid request, unresolved scope, conflicting authority configuration, and unavailable authority without
  permitted degraded handling return service errors with fixed `error_code` semantics instead of business decisions.
- [ ] A licensing scope cannot be configured as authoritative in both legacy and native modes simultaneously, and no
  request silently falls back from one authority to another after routing has been resolved.
- [ ] Legacy-authoritative and native-authoritative scopes can coexist in the same deployment through explicit routing,
  and the same normalized licensing scope resolves to the same authority mode consistently across routing, cache
  identity, and audit records.
- [ ] When `resolved_authority_mode = disabled`, decision-producing operations return `decision_outcome = deny` with
  primary `reason_code = scope_disabled` rather than a service error.
- [ ] Degraded decisions are explicitly marked `decision_state = degraded`, are never reported as authoritative, and are
  allowed only on read / preview surfaces that are in scope for the active priority tier; p1 state-changing operations
  return service errors instead of degraded success results when fresh authoritative evaluation is unavailable.
- [ ] Decision responses identify the resolved authority clearly enough to distinguish entitlement-driven denial from
  quota-driven denial or restriction.
- [ ] The legacy VendorA adapter is implemented behind Licensing Service public contracts, and platform consumers cannot
  bypass the module to consume adapter behavior directly.
- [ ] The quota layer does not independently take ownership of `entitlement_existence`, `feature_enablement`,
  `effective_licensed_limit`, or `commercially_authoritative_consumed_quantity`.
- [ ] Audit and telemetry records for a decision can be joined by `correlation_id` and show authority mode, decision
  outcome, decision state, primary reason code, and freshness state.
- [ ] The module separates fast runtime enforcement from durable authoritative accounting for the lifecycle states
  delivered in p1.

#### Deferred Acceptance (p2/p3)

- [ ] Later-tier decision fields such as `policy_effect`, `contributing_reason_codes`, structured explanation,
  `admission_outcome`, `retry_after_seconds`, `admitted_magnitude`, `target_policy_ids`, `release_outcome`, and
  `renewable_until` become acceptance criteria only in the priority tiers declared by their owning FRs.
- [ ] Later-tier runtime surfaces such as `Release`, `Update`, `GetEffectiveQuota`, `GetUsageSummary`, and lease
  operations become acceptance criteria only in the priority tiers declared by their owning FRs.
- [ ] Reconciliation, released-state accounting, shared recovery state machines, absolute stale-write protection,
  hierarchical policy resolution, shadow comparison, subject-kind registry enforcement, and full deterministic replay
  are not p1 acceptance gates; once delivered, they **MUST** satisfy the semantics of their owning FRs and ADRs.
- [ ] Consumer modules, operators, and migration engineers gain the broader target-state capabilities of soft outcomes,
  effective quota inspection, usage summaries, reconciliation visibility, shadow comparison, and extensible governed
  dimensions only as the document advances through p2 and p3.

## 11. Assumptions

The assumptions below are intentionally biased toward the p1 delivery contract while preserving room for future
evolution.

- Some production environments will continue to depend on legacy VendorA licensing for selected scopes during
  transition.
- Not all modules need native-authoritative licensing on day one.
- The platform can maintain canonical projections or cached views of authority data sufficient for p1 online checks;
  richer read models may be added in later tiers.
- Operators can manage authority-routing configuration per licensing scope.
- p1 deployment is single-region even though the module must preserve a migration path toward future multi-region
  operation.
- Consumer modules provide stable subject identifiers, scope context, action metadata, a `client_operation_id` for
  business correlation, and per-call `idempotency_key` values for the p1 reserve and settlement flows.
- Consumer modules enforce returned p1 quota outcomes close to execution and adopt later-tier module semantics only as
  those surfaces are delivered.
- Billing and invoicing remain outside the module responsibility boundary.
- Calendar-based quota periods use the closed set `{hourly, daily, weekly, monthly, billing_cycle}` with UTC-aligned
  window boundaries; window intervals are half-open `[window_start, window_end)` and an event at `window_end` belongs to
  the next window. `weekly` is anchored to ISO 8601 Monday `00:00:00 UTC`; `monthly` is anchored to the 1st of the month
  at `00:00:00 UTC`; `billing_cycle` is anchored by a per-policy `billing_anchor_day`. The normative policy-admission
  validation of `period` and `billing_anchor_day` (including the `[1, 28]` range rule for `billing_anchor_day`) is
  specified under `cpt-cf-licensing-service-fr-quota-policy-lifecycle`; this assumption bullet is descriptive of
  operator-visible semantics and defers to that FR for binding force. Sliding-window and token-bucket rate-limit shapes
  are out of p1 scope and may be added in a future version through additive, non-breaking extension of `policy.period`.

## 12. Risks

Risks below cover both the p1 operating contract and later expansion. Rows involving shadowing, reconciliation, or
highly extensible governance become materially more important in p2/p3 than in the initial rollout.

| Risk                                                             | Impact                                                                                  | Mitigation                                                                                                             |
|------------------------------------------------------------------|-----------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------|
| Legacy adapter mapping drift                                     | Wrong canonical entitlements or limits                                                  | Contract tests and shadow comparison metrics                                                                           |
| Projection staleness                                             | Incorrect or delayed decisions                                                          | Freshness metadata, explicit degraded mode, reconciliation                                                             |
| Dual-authority misconfiguration                                  | Conflicting business truth                                                              | Strict validation and one-authority-per-scope rules                                                                    |
| Internal layer ownership confusion                               | Double counting or under-enforcement                                                    | Explicit usage-class ownership matrix                                                                                  |
| Feature metadata drift                                           | Quota layer interprets effective limits with the wrong lifecycle or retention semantics | Version feature metadata, use contract tests, and run integration tests                                                |
| Missing operation identity propagation                           | Difficult recovery analysis across preview, reservation, and settlement                 | Preserve both `client_operation_id` and `correlation_id` across telemetry                                              |
| Migration complexity                                             | Slow rollout and operator uncertainty                                                   | Shadow mode, priority-tiered authority routing, detailed observability                                                 |
| Hot-key contention on popular tenant, subject, or usage-dimension scopes | Elevated decision latency and uneven enforcement behavior                               | Partition fast-path state by scope and monitor high-cardinality contention patterns                                    |
| Consumer modules integrate reserve and settlement inconsistently | Quota drift, double charging, or under-enforcement                                      | Provide stable SDK contracts, idempotent semantics, and integration tests for consumer patterns                        |
| Reconciliation lag grows during outages or event backlog         | Remaining quota becomes stale and valid work is blocked                                 | Instrument backlog health, alert on reconciliation delay, and provide expiry-driven recovery paths                     |
| Uncontrolled custom resource type growth creates semantic drift  | Policy sprawl and operator confusion                                                    | Govern resource type registration with metadata standards and validation rules                                         |
| Module size and complexity                                       | Slower development velocity as entitlement + quota logic coexist                        | Internal layering, clear ownership matrix, and split into separate modules if module exceeds maintainability threshold |

## 13. Open Questions

Unresolved questions that need answers before or during implementation.

No affected product family **MUST** be implemented using undocumented defaults for these questions; each affected
product family requires an approved extension to this PRD or equivalent requirements artifact before implementation.

- What product-family-specific degraded-policy values and freshness thresholds should be configured for the read-only
  operation classes already declared degraded-eligible in this PRD?
- Which product families require an explicit per-policy override of `max_deferred_retry_after_seconds` away from the
  module-wide default of `604_800` seconds, and which ones should stay on the default profile?
- Which additional product-family-specific usage classes, beyond the mandatory baseline ownership matrix in this PRD,
  must be explicitly classified before those product families are onboarded?
- What is the final v1 resource extensibility model for custom resource types: centrally registered, module-namespaced,
  or hybrid?
- Which classes of operations default to explicit deny versus controlled emergency allow when quota dependencies are
  degraded?
- What bounded overshoot tolerances are accepted for each quota class in single-region and future multi-region
  deployments?
- Which policy authoring boundaries belong to platform operators versus tenant administrators in v1?
- Where is the final boundary between authoritative quota accounting and downstream billing or chargeback systems?

## 14. Industry Reference Patterns

### Envoy Rate Limit / Rate Limit Quota Service (RLQS)

Envoy separates request-time enforcement from centralized quota state through dedicated rate-limit and quota protocols.
The pattern demonstrates that admission decisions can be delegated to a specialized quota component while enforcement
remains at the edge or service boundary.

### Google Cloud Quotas

Google Cloud exposes quotas as governed limits over multiple dimensions such as API calls, resource capacity, and
regional allocations, with clear project and service scoping. The pattern demonstrates hierarchical scope resolution,
operator-visible quota introspection, and the distinction between quota governance and billing.

### Kubernetes ResourceQuota

Kubernetes ResourceQuota constrains aggregate resource consumption within a namespace across multiple resource kinds and
rejects operations that would exceed the allowed budget. The pattern demonstrates multi-dimensional admission control,
hard-limit enforcement, and aggregation at a scoped boundary.

### OpenStack Unified Limits

OpenStack Unified Limits centralizes registered limits and scoped enforcement semantics across services while keeping
service-specific execution local. The pattern demonstrates centralized policy registration, reusable limit definitions,
and explicit scope-aware enforcement contracts.
