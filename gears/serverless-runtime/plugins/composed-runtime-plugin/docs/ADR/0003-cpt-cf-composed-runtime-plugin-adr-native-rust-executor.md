**ID**: `cpt-cf-composed-runtime-plugin-adr-native-rust-executor`

## Context and Problem Statement

Some callables need to run as native Rust code in-process: high-throughput data transforms, callables that wrap native libraries (numerics, parsing, crypto), platform-shipped utility functions, and customer-supplied performance-sensitive code. Starlark ([ADR-0002](0002-cpt-cf-composed-runtime-plugin-adr-starlark-runtime.md)) is intentionally sandboxed and not the right fit for these. The DSL-on-Temporal plugin ([serverless-runtime ADR-0003](../../../../docs/ADR/0003-cpt-cf-serverless-runtime-adr-workflow-dsl.md), [serverless-runtime ADR-0004](../../../../docs/ADR/0004-cpt-cf-serverless-runtime-adr-temporal-workflow-engine.md)) is a separate fat plugin under [serverless-runtime ADR-0005](../../../../docs/ADR/0005-cpt-cf-serverless-runtime-adr-thin-host.md) and is for declarative orchestration, not for the native action implementations the orchestration calls.

Native callables also need to be deployable and updatable without restarting the host process. Operators want to ship new versions, hot-fix a faulty implementation, or roll a tenant onto a new build, without taking the platform down. This rules out compile-time linking as the only mechanism.

This ADR defines the **native Rust function executor** as one of the embedded executors hosted inside the Composed Runtime plugin from [ADR-0001](0001-cpt-cf-composed-runtime-plugin-adr-composed-runtime.md). The executor packages native callables as **hot-loadable dynamic libraries** (`.so` / `.dylib` / `.dll`) with **plugin-managed state**, so a library can be unloaded, replaced, and reloaded cleanly while the host process continues serving traffic. The same library can also be served from the Composed Runtime plugin's managed-out-of-process variant — the in-plugin router from ADR-0001 makes the deployment mode transparent to callers. Each registered native callable is surfaced to the host as a `FunctionHandler<I, O>` / `WorkflowHandler<I, O>` SDK impl auto-generated by the SDK proc-macro (`#[native_callable]`) and wrapping the executor's invocation path.

The design draws on a parallel Go implementation the team already operates; this ADR translates that approach to Rust idioms and to ModKit's existing module / `ClientHub` / OoP infrastructure.

## Decision Drivers

* Native performance for callables that are too hot for an interpreter (ms or sub-ms budgets)
* Hot reload: replace a native callable's implementation without restarting the host process
* Clean unload: a reload must not leak threads, file handles, allocations, or background tasks; static and thread-local state must be either absent or fully owned by the Composed Runtime plugin
* Composability with fellow embedded executors via the in-plugin router from ADR-0001 (a native callable is just another `EmbeddedExecutor` inside the Composed Runtime plugin)
* Mode transparency: a native callable runs in-process by default and can be served from the Composed Runtime plugin's managed-out-of-process variant for isolation, without caller changes, using ModKit's OoP backend
* Reuse of ModKit primitives: `ClientHub`, `OopBackend`, `modkit-transport-grpc`, `DirectoryService` (see [ADR-0001](0001-cpt-cf-composed-runtime-plugin-adr-composed-runtime.md))
* ABI safety: the dynamic-library interface must survive Rust toolchain changes within a supported window, or the Composed Runtime plugin must reject incompatible libraries cleanly
* SDK conformance: each registered native callable must be surfaced to the host through `serverless-runtime-sdk`'s `FunctionHandler<I, O>` / `WorkflowHandler<I, O>` traits and emit timeline events through `ServerlessRuntimeClient`; library authors do not implement these SDK traits directly — the proc-macro generates them
* Replay correctness: a native callable participates in the same `ExecutionContext` checkpoint model as Starlark and any future embedded executor; reload must not break in-flight invocations
* Isolation escape hatch: untrusted or unstable native code (e.g. customer-supplied) must run in the plugin's managed-out-of-process variant for blast-radius containment

## Considered Options

* **Option A**: Compile-time plugin registry — each callable is a Rust crate built into the host binary
* **Option B**: WebAssembly (Wasm) — callables compiled to Wasm and loaded via a Wasm runtime
* **Option C**: Out-of-process only — every native callable is its own ModKit OoP module reached over gRPC
* **Option D**: Hot-loadable dynamic libraries with plugin-managed state, in-process by default and served from the Composed Runtime plugin's managed-OoP variant on opt-in, plugged into the in-plugin router from ADR-0001 — **chosen**

## Decision Outcome

Chosen option: **"Option D: Hot-loadable dynamic libraries with runtime-managed state"**.

Native callables are packaged as Rust `cdylib` artifacts. Each library exports a small, stable C-ABI registration entrypoint (`cf_serverless_register_v1`) that, when called by the Composed Runtime plugin's loader, returns a versioned descriptor table — per callable, its GTS ID, params/returns/checkpoint schema IDs, and an `invoke` function pointer, plus an optional library-level `on_unload` hook (see the ABI definition below). Suspend/resume are not separate ABI symbols — suspension is expressed through `invoke` (an `InvokeStatus::Pending` plus a wakeup token) and replay re-enters `invoke`. The plugin loads the library via `libloading`, validates the ABI version, and registers each declared callable into the Composed Runtime plugin's embedded-executor and callable registries from ADR-0001.

State ownership is the central design rule: **callable code is stateless across invocations from its own perspective**, with all per-invocation and per-callable state living in the Composed Runtime plugin, surfaced via the unified `ExecutionContext`. This makes reloads clean — there is no library-resident state that must be migrated. Background tasks, caches, and connection pools are plugin-managed services obtained through `ExecutionContext` (and ultimately backed by `ClientHub`-resolved interfaces); a library that needs them does not own them.

Hot reload is one ordered flow: (1) **load** the new library and validate its ABI version and schema compatibility; (2) **drain** the outgoing library — stop routing new invocations to it and let in-flight invocations checkpoint and complete or migrate; (3) **resume** any migrated invocations through their checkpoint envelopes against the new library; (4) **unload** the outgoing library once it has zero in-flight invocations. The reload boundary is between steps 2 and 3: an invocation's next resume after drain is served by the new library. The mechanics ride entirely on the Composed Runtime plugin's checkpoint store and callable registry from ADR-0001 — there is no library-to-library state handoff. (The detailed protocol is in [Hot Reload Protocol](#hot-reload-protocol).)

The managed-out-of-process variant is the same library run inside a child-process instance of the Composed Runtime plugin (the plugin's managed-OoP form) that exposes the embedded executor over `modkit-transport-grpc`. The in-plugin router resolves either the in-process loader or the managed-OoP gRPC client identically through `ClientHub`.

### Consequences

* Good, because native callables run at native speed in-process by default
* Good, because hot reload removes one of the main reasons operators previously reached for managed-OoP deployment
* Good, because the plugin-owned state model keeps reload semantics simple and verifiable: the only thing that needs to be "drained" is in-flight invocations, not memory state
* Good, because the same library binary works in-process and in the plugin's managed-OoP variant — deployment mode is configuration, not a code fork
* Good, because composability with Starlark and any other embedded executor is automatic via the in-plugin router from ADR-0001; callables hosted in other plugins (e.g. DSL/Temporal) are not reachable via `ctx.invoke` — cross-plugin composition is a host-orchestration concern, see [ADR-0001](0001-cpt-cf-composed-runtime-plugin-adr-composed-runtime.md) Non-goals
* Good, because tenant scoping, tracing, and replay are inherited from the Composed Runtime plugin (ADR-0001), not re-implemented per library
* Bad, because the registration ABI is a stable interface; changes need versioning and a deprecation window
* Bad, because in-process loading inherits the host's memory space — a buggy library can corrupt heap or panic the process; this is precisely why the managed-OoP escape hatch exists for untrusted code
* Bad, because Rust does not have a stable ABI; the registration table is C-ABI and exchanges only POD types and function pointers, with payloads passed as bytes — slightly more friction than a pure-Rust trait at the boundary
* Bad, because hot reload requires careful symbol management on platforms with `RTLD_LOCAL` quirks; mitigated by `libloading` and a small platform-specific shim

### Confirmation

* Build a sample `cdylib` declaring two callables; load it; invoke each via the in-plugin router from ADR-0001; confirm trace, replay, tenant scoping, and `ExecutionContext::invoke` of another embedded callable all work.
* Reload test: load v1, drive traffic, ship v2 with a different implementation of one callable, drain, reload, resume in-flight invocations from their checkpoint envelopes; confirm no leaked threads / file descriptors / mappings (verified via `/proc/self/maps` and a leak detector).
* Managed-OoP test: ship the same `cdylib` inside the Composed Runtime plugin's managed-OoP child instance; configure callables for managed-OoP; confirm caller code is identical and end-to-end behavior matches in-process.
* ABI rejection test: load a library built against an older registration ABI version; confirm clean refusal and a clear diagnostic.
* Crash containment test (managed-OoP mode): induce a panic in the library; confirm the host process and other tenants are unaffected; the OoP backend restarts the child plugin instance; affected invocations resume.
* Performance test: in-process invoke latency p95 < 5µs (function call + minor router overhead); managed-OoP invoke latency dominated by gRPC RTT.

## Pros and Cons of the Options

### Option A: Compile-time plugin registry

* Good, because simplest possible model — direct Rust trait calls, full type system, no ABI concerns
* Good, because no runtime loader, no `libloading`
* Bad, because every change to a native callable requires rebuilding and redeploying the host
* Bad, because customer-supplied or third-party callables cannot be added without a host build
* Bad, because no isolation story: a panic in any callable takes down the host

### Option B: WebAssembly (Wasm)

* Good, because strong sandboxing and a stable, portable binary format
* Good, because hot reload is natural (load a new module, swap)
* Good, because cross-language authoring is possible (Rust, Go, AssemblyScript, …)
* Bad, because performance is worse than native — defeats the main reason this executor exists
* Bad, because host-call surface is heavier (Wasm imports, marshaling) than a Rust function call
* Bad, because Wasm is a credible future addition as its own ADR; this executor is specifically for callables that need native speed and native crate access

### Option C: Out-of-process only

* Good, because uniform isolation; one model to operate
* Good, because reloads are just process restarts
* Bad, because every native callable pays gRPC RTT — wipes out the performance motivation
* Bad, because forces multi-process deployment for the simplest local-dev case
* Bad, because the Composed Runtime plugin already supports a managed-OoP variant per [ADR-0001](0001-cpt-cf-composed-runtime-plugin-adr-composed-runtime.md) — there is no need to make it the only option here

### Option D: Hot-loadable dynamic libraries — chosen

* Good, because in-process performance is preserved
* Good, because hot reload eliminates one of the main pressures toward OoP-by-default
* Good, because the same artifact deploys in-process or in the Composed Runtime plugin's managed-OoP variant via the mode-transparent router from ADR-0001
* Good, because plugin-managed state makes unload semantics simple and auditable
* Good, because the design is straightforwardly portable from a parallel Go implementation
* Bad, because the registration ABI is now a long-term commitment
* Bad, because in-process loading is not a sandbox; trusted callables only, with managed-OoP for the rest

## More Information

### Goals

- Provide a way to ship new native Rust callables, or new versions of existing ones, without restarting the host process.
- Keep native callable code stateless across invocations; the Composed Runtime plugin owns all persistent and per-invocation state.
- Plug seamlessly into the in-plugin router from ADR-0001; native callables are dispatched the same way Starlark and any other embedded callables are.
- Make in-process and managed-OoP deployment a configuration choice, not a code change.

### Non-goals

- Cross-language native callables (Wasm or other foreign-language plugins). A future Wasm embedded executor is a separate ADR.
- A full plugin marketplace / signing / distribution model. This ADR specifies the loader contract; signing and distribution are operational concerns deferred to a later ADR.
- Defining the unified `ExecutionContext` surface (specified in ADR-0001).
- Specifying the Temporal plugin or the Starlark embedded executor (serverless-runtime ADR-0004, ADR-0002).

### Library Layout

> **Note:** The `cf-serverless-native-sdk` crate referenced below does not exist yet. The name is used here as the proposed identifier for the SDK that this ADR introduces; it will be created as part of implementing this decision.

A native callable library is a Rust `cdylib`:

```toml
[package]
name = "example-native-callables"
version = "0.1.0"

[lib]
crate-type = ["cdylib"]

[dependencies]
cf-serverless-native-sdk = "0.1"   # this ADR's SDK crate; re-exports the registration ABI
```

The SDK crate `cf-serverless-native-sdk` provides:
- the `#[native_callable]` proc-macro that derives, for each annotated Rust function: (a) the C-ABI registration entry consumed by the loader, (b) a `FunctionHandler<I, O>` (or `WorkflowHandler<I, O>` for `workflow_traits`-bearing callables) impl that the Composed Runtime plugin surfaces to the host through `RuntimeAdapter`, and (c) wiring from `EmbeddedExecutorError` to `ServerlessSdkError`
- a typed `ExecutionContext` wrapper around the C-ABI surface (`Ctx`), so library code writes ordinary `async fn invoke(ctx: &Ctx, input: Input) -> Result<Output>`; `Ctx::sdk_context()` and `Ctx::environment()` re-expose the SDK's `Context` and `Environment` for code that needs them
- the `cf_serverless_register_v1` exported symbol assembled from all `#[native_callable]` declarations

### Registration ABI (C-ABI, versioned)

```c
// Version exposed by the library; runtime checks against supported range.
extern uint32_t cf_serverless_native_abi_version(void);

// Returns a descriptor table. The runtime owns the table after this call.
extern const NativeRegistration* cf_serverless_register_v1(const HostInterface* host);

struct NativeRegistration {
    uint32_t abi_version;
    uint32_t callable_count;
    const NativeCallableDescriptor* callables;
    void (*on_unload)(void);   // optional; runtime invokes before dlclose
};

struct NativeCallableDescriptor {
    const char* callable_id;          // GTS ID, e.g. "gts.x.core.serverless.function.v1~vendor.app.compose.v1~"
    const char* params_schema_id;     // GTS schema id
    const char* returns_schema_id;    // GTS schema id
    const char* checkpoint_schema_id; // GTS schema id, or null if stateless

    // invoke: runtime passes a context handle and a JSON-bytes input;
    // library returns a JSON-bytes output via the out-pointer.
    // Async is expressed by returning Pending and a wakeup token; details in the SDK.
    InvokeStatus (*invoke)(
        const ContextHandle* ctx,
        const Bytes* input,
        Bytes* output,
        Error* err);
};
```

Library code never holds a `&'static` reference to the runtime; the SDK wrapper keeps the host pointer scoped to the call.

### Plugin-Managed State

A native callable is given an `ExecutionContext` per invocation. State that must outlive an invocation is checkpointed via `ctx.write_checkpoint(label, schema_id, payload)`. State that must outlive the library (caches, pools, clients) is obtained via `ctx.host_service::<dyn SomeService>()`, which resolves through `ClientHub`.

Concretely, the contract on library authors is:

- **No library-resident static state** that survives a reload. `lazy_static!` / `OnceCell` / global `Mutex` are forbidden in callable code; the SDK proc-macro lints for these.
- **No background threads or tasks.** If a callable needs scheduling, it goes through the Composed Runtime plugin's job system.
- **No file handles or connections held across invocations.** Connection pools are plugin-provided services, not library properties.
- **`on_unload`** is for releasing the small table data (descriptor strings) the library allocated at register time, nothing else.

These rules make reload a structural property: dropping the library is safe because there is nothing to drop except the registration table.

### Hot Reload Protocol

1. **Announce.** Operator submits library `v_new` via the Composed Runtime plugin's control API.
2. **Load shadow.** The plugin loads `v_new` into a new `Library` slot, calls `cf_serverless_native_abi_version` and `cf_serverless_register_v1`, validates schema compatibility (callable IDs, params/returns schemas).
3. **Drain.** For each callable present in both `v_old` and `v_new`, the in-plugin router stops routing new invocations to `v_old` and starts routing them to `v_new`. In-flight invocations on `v_old` continue using their existing context; if they suspend (snapshot), their next resume goes to `v_new`. The drain is bounded by a configurable timeout; remaining invocations are cancelled (their compensation runs through `ExecutionContext`).
4. **Unload.** Once `v_old` has zero in-flight invocations, the plugin calls `on_unload` and `dlclose` on `v_old`.
5. **Done.** `v_new` is the only library serving traffic.

For the plugin's managed-OoP variant, hot reload is a child-process restart: the managed-OoP plugin instance loads `v_new` on startup; the in-plugin router moves traffic via the OoP backend's existing supervisor.

### Mode Transparency

A native callable's deployment mode is declared in its registration metadata or by tenant policy. In `in_process` mode the executor is the in-process loader; in `managed_oop_*` mode the executor is the same loader running inside the Composed Runtime plugin's child-process instance, exposed via `modkit-transport-grpc` and resolved through `ClientHub`. Caller code never branches on mode (per ADR-0001).

### Replay and Composition

The native executor is an `EmbeddedExecutor` per ADR-0001. It receives the unified `ExecutionContext`. A native callable may:
- read prior checkpoints (`ctx.read_checkpoint`) — useful for callables that participate in long-running orchestrations
- write checkpoints (`ctx.write_checkpoint`) under its own GTS schema
- invoke fellow embedded callables (`ctx.invoke`) — the in-plugin router routes to Starlark, other native callables, or future embedded executors (CEL, etc.), in-process or in the plugin's managed-OoP variant. A callee hosted in a different plugin (e.g. a DSL/Temporal callable) is **not** reachable via `ctx.invoke`; cross-plugin composition is arranged at the host orchestration layer, not through this router (see [ADR-0001](0001-cpt-cf-composed-runtime-plugin-adr-composed-runtime.md) Non-goals).

This is what makes the user requirement "a user may write a code workflow and use Starlark and hot-loaded native Rust functions interchangeably inside the Composed Runtime plugin" hold uniformly.

### Reuse of ModKit Primitives

| Concern | ModKit primitive |
|---------|------------------|
| Embedded executor registration inside the Composed Runtime plugin | `ClientHub` (`libs/modkit/src/client_hub.rs`) under `dyn EmbeddedExecutor` with `ClientScope::gts_id(callable_id)` |
| Managed-OoP child-process supervision | `OopBackend` / `LocalProcessBackend` / K8s backend (`libs/modkit/src/backends/`) |
| Managed-OoP bootstrap | `bootstrap::oop::run_oop_with_options` (`libs/modkit/src/bootstrap/oop.rs`) |
| Managed-OoP gRPC transport | `modkit-transport-grpc` (`libs/modkit-transport-grpc/`) |
| Module discovery | `DirectoryService` (`libs/modkit/src/directory.rs`) |
| Configuration | standard ModKit module config |

The native executor adds, on top: the `cdylib` registration ABI, the `libloading`-based loader, the SDK crate (`cf-serverless-native-sdk`), and the hot-reload drain protocol.

### Relationship to Other ADRs

- **[serverless-runtime ADR-0001](../../../../docs/ADR/0001-cpt-cf-serverless-runtime-adr-callable-type-hierarchy.md):** Native callables register as `Function` or `Workflow` per the type model. Workflow-shaped native callables set `workflow_traits` and use checkpoint envelopes for suspension.
- **[serverless-runtime ADR-0005](../../../../docs/ADR/0005-cpt-cf-serverless-runtime-adr-thin-host.md) (Thin host, fat runtime plugins):** Grandparent decision. The Composed Runtime plugin (which hosts this executor) is one runtime plugin under that model.
- **[ADR-0001](0001-cpt-cf-composed-runtime-plugin-adr-composed-runtime.md):** Parent decision. The native executor is one of the embedded executors inside the Composed Runtime plugin: it implements `EmbeddedExecutor` and reuses the in-plugin router, `ExecutionContext`, and managed-OoP infrastructure defined there. Hot reload is built on top of ADR-0001's checkpoint store; it is not a separate state-migration mechanism.
- **[ADR-0002](0002-cpt-cf-composed-runtime-plugin-adr-starlark-runtime.md):** Fellow embedded executor inside the same Composed Runtime plugin. Starlark code can call native callables via `ctx.invoke`, and native code can invoke Starlark callables via `ctx.invoke`. The in-plugin router resolves transparently across embedded executors and across the in-process / managed-OoP boundary.
- **[serverless-runtime ADR-0003](../../../../docs/ADR/0003-cpt-cf-serverless-runtime-adr-workflow-dsl.md), [serverless-runtime ADR-0004](../../../../docs/ADR/0004-cpt-cf-serverless-runtime-adr-temporal-workflow-engine.md):** Separate fat plugin. Composition between a DSL/Temporal workflow and a native callable in this plugin, if supported, is arranged at the host orchestration layer — not through this plugin's in-plugin router and not through `ctx.invoke`.

* `cpt-cf-serverless-runtime-nfr-tenant-isolation` — the Composed Runtime plugin's managed-OoP variant (inherited from ADR-0001) provides OS-level isolation for untrusted native callables
